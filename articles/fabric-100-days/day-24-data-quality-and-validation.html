Why Data Quality Is the Silent Killer of Analytics Projects

Imagine spending six months building a beautiful Power BI semantic model on Microsoft Fabric, only to have your CFO point out that the revenue figures don't match the ERP system — by 12%. That meeting ends careers. Data quality failures are rarely dramatic; they accumulate silently across pipelines, transformations, and ingestion jobs until they surface at the worst possible moment: a board presentation, a regulatory audit, or a customer-facing dashboard.

In Microsoft Fabric, data flows through multiple layers — from OneLake raw storage through Bronze/Silver/Gold Delta Parquet tables, into semantic models consumed via DirectLake mode. Each layer is a potential point of corruption, duplication, schema drift, or silent null propagation. Unlike traditional data warehouses where a single ETL team controlled every transformation, Fabric's open ecosystem means notebooks, Dataflows Gen2, pipelines, and even external writers can all touch the same Delta tables simultaneously.

This article gives you a systematic, production-grade approach to data quality and validation in Microsoft Fabric — covering frameworks, code patterns, native features, and the non-obvious pitfalls that trip up even experienced engineers.

Core Concepts in Data Quality

The Six Dimensions of Data Quality

Before writing a single line of validation code, you need a shared vocabulary. Data quality is not a binary state — it has six measurable dimensions, each requiring different detection and remediation strategies:

Dimension Definition Example in Fabric Context Detection Method
Completeness Required fields are populated customer_id is NULL in 3% of orders rows in Silver Delta table NULL count assertions in notebook
Accuracy Values reflect real-world truth Unit price of $0.001 for a product that costs $499 Range checks, cross-system reconciliation
Consistency Same entity has same values across systems Customer name "Acme Corp" in CRM vs "ACME CORPORATION" in ERP Cross-table joins, fuzzy matching
Timeliness Data is available when needed Sales data from yesterday still not in Gold layer at 8 AM Watermark checks, pipeline monitoring
Uniqueness No unintended duplicates exist Same invoice loaded twice due to pipeline retry logic Duplicate key detection on Delta tables
Validity Values conform to defined formats/rules Email field contains "N/A" or phone field has letters Regex pattern matching, domain lookups

Where Validation Fits in the Medallion Architecture

In Fabric's Medallion architecture, validation is not a single checkpoint — it is a continuous contract between layers. The key insight most teams miss is that Bronze should accept everything but log everything, Silver should enforce structural contracts, and Gold should enforce business contracts. Mixing these responsibilities causes either data loss (rejecting too early) or silent corruption (validating too late).

  • Bronze layer: Schema-on-read, raw Delta Parquet files in OneLake. Validate only that files arrived, row counts are non-zero, and source timestamps are present. Never reject records here.
  • Silver layer: Apply structural validations — data types, NULL constraints on key columns, referential integrity checks against dimension tables. Quarantine failing records to a _quarantine Delta table rather than dropping them.
  • Gold layer: Apply business rule validations — revenue must be positive, order dates cannot be in the future, margin percentages must be between -100% and 500%. Fail the pipeline if Gold-level violations exceed a defined threshold (e.g., 0.1% of daily volume).

Validation Approaches in Microsoft Fabric

Approach 1: PySpark Notebook-Based Validation Framework

The most flexible and auditable approach is a reusable PySpark validation framework running in Fabric Notebooks. The following pattern implements a declarative rule engine that produces a structured quality report written back to OneLake as a Delta table — making quality metrics queryable in Power BI.

from pyspark.sql import DataFrame
from pyspark.sql import functions as F
from pyspark.sql.types import StructType, StructField, StringType, LongType, DoubleType, TimestampType
from datetime import datetime
from typing import List, Dict, Any

# ── Validation Rule Engine ──────────────────────────────────────────────────

def check_null_rate(df: DataFrame, column: str, threshold: float = 0.0) -> Dict[str, Any]:
    """Returns PASS/FAIL if null rate exceeds threshold (0.0 = no nulls allowed)."""
    total = df.count()
    null_count = df.filter(F.col(column).isNull()).count()
    null_rate = null_count / total if total > 0 else 0.0
    return {
        "rule": f"null_rate:{column}",
        "dimension": "completeness",
        "column": column,
        "metric_value": round(null_rate, 6),
        "threshold": threshold,
        "status": "PASS" if null_rate <= threshold else "FAIL",
        "failing_records": null_count
    }

def check_uniqueness(df: DataFrame, key_columns: List[str]) -> Dict[str, Any]:
    """Detects duplicate rows based on composite key."""
    total = df.count()
    distinct = df.select(key_columns).distinct().count()
    duplicate_count = total - distinct
    return {
        "rule": f"uniqueness:{'|'.join(key_columns)}",
        "dimension": "uniqueness",
        "column": "|".join(key_columns),
        "metric_value": round(duplicate_count / total, 6) if total > 0 else 0.0,
        "threshold": 0.0,
        "status": "PASS" if duplicate_count == 0 else "FAIL",
        "failing_records": duplicate_count
    }

def check_value_range(df: DataFrame, column: str, min_val: float, max_val: float) -> Dict[str, Any]:
    """Validates numeric column falls within [min_val, max_val]."""
    out_of_range = df.filter(
        (F.col(column) < min_val) | (F.col(column) > max_val)
    ).count()
    total = df.count()
    return {
        "rule": f"range:{column}[{min_val},{max_val}]",
        "dimension": "validity",
        "column": column,
        "metric_value": round(out_of_range / total, 6) if total > 0 else 0.0,
        "threshold": 0.001,  # Allow up to 0.1% anomalies
        "status": "PASS" if out_of_range / total <= 0.001 else "FAIL",
        "failing_records": out_of_range
    }

def run_validation_suite(df: DataFrame, table_name: str, rules: List[Dict]) -> DataFrame:
    """Executes all rules and writes results to DQ log table."""
    results = []
    run_ts = datetime.utcnow()
    
    for rule in rules:
        rule_type = rule["type"]
        if rule_type == "null_rate":
            result = check_null_rate(df, rule["column"], rule.get("threshold", 0.0))
        elif rule_type == "uniqueness":
            result = check_uniqueness(df, rule["key_columns"])
        elif rule_type == "range":
            result = check_value_range(df, rule["column"], rule["min"], rule["max"])
        else:
            continue
        
        result["table_name"] = table_name
        result["run_timestamp"] = run_ts
        results.append(result)
    
    schema = StructType([
        StructField("table_name", StringType()),
        StructField("rule", StringType()),
        StructField("dimension", StringType()),
        StructField("column", StringType()),
        StructField("metric_value", DoubleType()),
        StructField("threshold", DoubleType()),
        StructField("status", StringType()),
        StructField("failing_records", LongType()),
        StructField("run_timestamp", TimestampType())
    ])
    
    result_df = spark.createDataFrame(results, schema=schema)
    
    # Write to OneLake DQ log table (append mode for historical tracking)
    result_df.write.format("delta").mode("append").saveAsTable("silver.dq_validation_log")
    
    return result_df

# ── Execute Against Silver Orders Table ────────────────────────────────────

orders_df = spark.read.format("delta").table("silver.orders")

rules = [
    {"type": "null_rate",   "column": "customer_id",  "threshold": 0.0},
    {"type": "null_rate",   "column": "order_date",    "threshold": 0.0},
    {"type": "null_rate",   "column": "order_amount",  "threshold": 0.005},
    {"type": "uniqueness",  "key_columns": ["order_id"]},
    {"type": "range",       "column": "order_amount",  "min": 0.01, "max": 1_000_000},
    {"type": "range",       "column": "quantity",      "min": 1,    "max": 10_000},
]

results_df = run_validation_suite(orders_df, "silver.orders", rules)
results_df.show(truncate=False)

# Fail the notebook if any FAIL results exist (stops downstream pipeline activities)
fail_count = results_df.filter(F.col("status") == "FAIL").count()
if fail_count > 0:
    raise ValueError(f"Data quality validation failed: {fail_count} rule(s) violated. Check silver.dq_validation_log.")

A non-obvious insight here: by writing results to silver.dq_validation_log as a Delta table with append mode, you accumulate a historical quality time series. You can then build a Power BI report showing quality trends over time — turning DQ from a gate into an observable metric. This table is also queryable via the Fabric SQL Analytics Endpoint without any additional configuration.

Approach 2: T-SQL Validation in the Warehouse

For teams working in Fabric Data Warehouse (not Lakehouse), T-SQL assertions are often faster to write and easier for SQL-native analysts to maintain. The following pattern uses a stored procedure approach with a results table:

-- Create DQ results table (run once)
CREATE TABLE dbo.dq_validation_results (
    validation_id    INT           IDENTITY(1,1) PRIMARY KEY,
    run_timestamp    DATETIME2     DEFAULT SYSUTCDATETIME(),
    table_name       NVARCHAR(128),
    rule_name        NVARCHAR(256),
    dimension        NVARCHAR(64),
    status           NVARCHAR(10),  -- 'PASS' or 'FAIL'
    failing_records  BIGINT,
    total_records    BIGINT,
    failure_rate     DECIMAL(10,6)
);

-- Validation procedure for Gold.FactSales
CREATE OR ALTER PROCEDURE dbo.usp_ValidateFactSales
AS
BEGIN
    SET NOCOUNT ON;
    
    DECLARE @total BIGINT = (SELECT COUNT(*) FROM gold.FactSales);
    DECLARE @run_ts DATETIME2 = SYSUTCDATETIME();

    -- Rule 1: No NULL SalesAmount
    INSERT INTO dbo.dq_validation_results 
        (table_name, rule_name, dimension, status, failing_records, total_records, failure_rate)
    SELECT
        'gold.FactSales',
        'null_check:SalesAmount',
        'completeness',
        CASE WHEN COUNT(*) = 0 THEN 'PASS' ELSE 'FAIL' END,
        COUNT(*),
        @total,
        CAST(COUNT(*) AS DECIMAL(10,6)) / NULLIF(@total, 0)
    FROM gold.FactSales
    WHERE SalesAmount IS NULL;

    -- Rule 2: SalesAmount must be positive
    INSERT INTO dbo.dq_validation_results 
        (table_name, rule_name, dimension, status, failing_records, total_records, failure_rate)
    SELECT
        'gold.FactSales',
        'range_check:SalesAmount[0.01,9999999]',
        'validity',
        CASE WHEN COUNT(*) = 0 THEN 'PASS' ELSE 'FAIL' END,
        COUNT(*),
        @total,
        CAST(COUNT(*) AS DECIMAL(10,6)) / NULLIF(@total, 0)
    FROM gold.FactSales
    WHERE SalesAmount <= 0 OR SalesAmount > 9999999;

    -- Rule 3: OrderDateKey must exist in DimDate (referential integrity)
    INSERT INTO dbo.dq_validation_results 
        (table_name, rule_name, dimension, status, failing_records, total_records, failure_rate)
    SELECT
        'gold.FactSales',
        'ref_integrity:OrderDateKey->DimDate',
        'consistency',
        CASE WHEN COUNT(*) = 0 THEN 'PASS' ELSE 'FAIL' END,
        COUNT(*),
        @total,
        CAST(COUNT(*) AS DECIMAL(10,6)) / NULLIF(@total, 0)
    FROM gold.FactSales fs
    LEFT JOIN gold.DimDate d ON fs.OrderDateKey = d.DateKey
    WHERE d.DateKey IS NULL;

    -- Rule 4: No duplicate SalesOrderLineKey
    INSERT INTO dbo.dq_validation_results 
        (table_name, rule_name, dimension, status, failing_records, total_records, failure_rate)
    SELECT
        'gold.FactSales',
        'uniqueness:SalesOrderLineKey',
        'uniqueness',
        CASE WHEN SUM(dup_count) = 0 THEN 'PASS' ELSE 'FAIL' END,
        SUM(dup_count),
        @total,
        CAST(SUM(dup_count) AS DECIMAL(10,6)) / NULLIF(@total, 0)
    FROM (
        SELECT SalesOrderLineKey, COUNT(*) - 1 AS dup_count
        FROM gold.FactSales
        GROUP BY SalesOrderLineKey
        HAVING COUNT(*) > 1
    ) dupes;

    -- Surface failures for pipeline to detect
    IF EXISTS (
        SELECT 1 FROM dbo.dq_validation_results 
        WHERE status = 'FAIL' 
          AND run_timestamp >= @run_ts
    )
    BEGIN
        RAISERROR('Data quality validation failed for gold.FactSales. Review dbo.dq_validation_results.', 16, 1);
    END
END;

-- Execute
EXEC dbo.usp_ValidateFactSales;

Native Fabric Features for Data Quality

Delta Lake Constraints

Delta Lake — the underlying format for all Fabric Lakehouse tables — supports CHECK constraints that enforce data quality at the storage layer. These are enforced on every write operation, including from external tools that write directly to OneLake. This is a powerful, often-overlooked capability:

  • Add constraints with: ALTER TABLE silver.orders ADD CONSTRAINT chk_positive_amount CHECK (order_amount > 0)
  • Constraints are stored in the Delta transaction log and enforced by the Delta writer — any write violating the constraint raises an error and the transaction is rolled back atomically
  • Constraints survive table clones and are visible in DESCRIBE DETAIL output
  • Limitation: Delta constraints do not support cross-table referential integrity — use application-layer checks for foreign key validation

Dataflows Gen2 Column Quality Profiling

Fabric Dataflows Gen2 (built on Power Query) includes built-in column profiling that shows null rates, distinct counts, and value distributions directly in the authoring UI. While this is excellent for exploration, it is not a substitute for automated validation — profiling results are not persisted or alertable. Use it for discovery, not enforcement.

Data Activator for Quality Alerting

Microsoft Fabric's Data Activator (formerly Reflex) can monitor your dq_validation_log Delta table and trigger alerts when FAIL records appear. Connect Data Activator to your DQ log table via the Fabric Real-Time Hub, define a trigger condition on status = 'FAIL', and route alerts to Teams or email. This closes the loop between detection and notification without custom alerting infrastructure.

Best Practices

Do Avoid
Write DQ results to a persistent Delta table so quality trends are queryable over time Logging DQ results only to notebook output — it disappears after the session ends
Use RAISE / RAISERROR to fail pipelines when critical rules are violated, blocking downstream Gold table refresh Logging failures silently and letting bad data flow into Gold — analysts will find it in a meeting
Quarantine failing records to a _quarantine Delta table with the original row plus a failure_reason column Dropping invalid records without a trace — you lose the ability to investigate root cause or recover data
Define DQ thresholds as configuration (e.g., a JSON config file in OneLake) so rules can be updated without code changes Hard-coding thresholds in notebook cells — changing a threshold requires a code review and deployment
Run lightweight row-count and watermark checks at the Bronze layer immediately after ingestion Waiting until Silver or Gold to detect that a source file was empty or truncated
Version your validation rules alongside your transformation code in Git (Fabric Git Integration) Maintaining validation rules in ad-hoc notebooks not connected to a Git repository
Use Delta table DESCRIBE HISTORY to correlate DQ failures with specific pipeline runs and committers Investigating DQ failures without checking the Delta transaction log — you're flying blind

Common Pitfalls

Pitfall Cause Fix
Validation passes but data is still wrong after pipeline retry Pipeline retried on failure, loading the same source data twice into a Delta table without deduplication logic, creating duplicates that pass uniqueness checks because the check ran before the retry Use Delta MERGE (upsert) instead of append for idempotent loads; run uniqueness checks after all retries complete
NULL checks pass on Silver but NULLs appear in Gold A JOIN in the Gold transformation introduces NULLs from unmatched rows (LEFT JOIN behavior), which were not present in Silver Add NULL checks on all Gold columns that result from JOIN operations, not just source columns
Row count validation passes but data is duplicated Checking only that row count is non-zero, not that it matches expected volume (e.g., yesterday's count ± 20%) Implement volume anomaly detection: compare today's row count to a rolling 7-day average and alert if deviation exceeds 30%
Delta CHECK constraints silently ignored by external writers Third-party tools writing Parquet files directly to OneLake storage paths bypass the Delta protocol and constraints entirely Enforce that all writes go through Spark or the Fabric SQL endpoint; use OneLake access policies to restrict direct storage writes
DQ notebook times out on F2 SKU for large tables Running full table scans for every validation rule on an F2 (2 CU) capacity with limited Spark resources Use Delta's OPTIMIZE and Z-ordering on key columns; use sampling (df.sample(0.1)) for non-critical checks; upgrade to F8+ for production DQ workloads
Schema drift breaks validation rules silently Source system adds a new column or renames an existing one; Bronze ingestion accepts it but Silver validation rules reference the old column name and throw a ColumnNotFound error that is caught and swallowed Add schema validation as the first rule in every suite: compare actual schema against a registered expected schema stored in a config Delta table

Real-World Scenario: Retail Sales Pipeline DQ Implementation

A retail company processes 2 million daily sales transactions from 400 stores into a Fabric Lakehouse. The finance team discovered a $3.2M discrepancy between the Gold layer revenue figure and the source ERP system. Here is the step-by-step remediation and DQ framework implementation:

  1. Audit the Delta transaction log: Run DESCRIBE HISTORY gold.FactSales to identify the pipeline run that introduced the discrepancy. Cross-reference the operationMetrics (numOutputRows) with the expected daily volume from the ERP extract manifest file stored in Bronze.
  2. Identify root cause: The Bronze ingestion pipeline used .mode("append") and retried three times due to a transient OneLake write timeout. Each retry appended the full day's 2M rows, resulting in 6M rows in Bronze and tripled revenue in Gold.
  3. Implement idempotent Bronze ingestion: Replace the append with a Delta MERGE keyed on transaction_id and store_id. Add a Bronze-layer row count check: after ingestion, assert that COUNT(*) WHERE load_date = today equals the row count in the source manifest file (written by the ERP extract job to a control table in Bronze).
  4. Add Silver-layer uniqueness validation: Deploy the PySpark validation framework above with a uniqueness rule on transaction_id. Wire the notebook into the Data Pipeline as an activity immediately after the Bronze-to-Silver transformation, before the Silver-to-Gold activity runs.
  5. Implement volume anomaly detection: Query the DQ log table to calculate a rolling 7-day average row count. If today's Silver row count deviates by more than 15% from the rolling average, fail the pipeline and trigger a Data Activator alert to the data engineering Teams channel.
  6. Add Gold-layer revenue reconciliation: Create a T-SQL stored procedure that compares the sum of SalesAmount in gold.FactSales for the current load date against the control total from the ERP manifest. A variance greater than 0.01% triggers a RAISERROR that blocks the Power BI dataset refresh.
  7. Build a DQ Power BI dashboard: Connect Power BI to the silver.dq_validation_log Delta table via DirectLake mode. Create a report showing daily pass/fail rates by dimension, trending over 90 days. Set up a Power BI data alert on the "FAIL count" measure to notify the data steward when failures occur.
  8. Document and version rules: Store all validation rule configurations as JSON files in a /config/dq_rules/ folder in OneLake, connected to the Fabric workspace Git repository. Any change to a threshold requires a pull request with approval from the data steward.
Outcome: After implementing this framework, the team detected a similar duplication issue three weeks later — within 4 minutes of pipeline completion, before any reports were refreshed. The finance team never saw bad data again. The DQ dashboard became a standing agenda item in the weekly data governance meeting.

Key Takeaways

  • Data quality is not a single checkpoint — it is a layered contract across Bronze (arrival), Silver (structural), and Gold (business) layers, each with different rules and failure modes.
  • Always quarantine invalid records to a _quarantine Delta table rather than dropping them — silent data loss is worse than a pipeline failure.
  • Delta Lake CHECK constraints enforce quality at the storage layer and apply to all writers, but they do not support cross-table referential integrity — you need application-layer checks for foreign key validation.
  • Write DQ results to a persistent Delta table to enable historical quality trending in Power BI — this transforms DQ from a gate into an observable, governable metric.
  • Volume anomaly detection (comparing today's row count to a rolling average) catches an entire class of bugs — empty files, duplicated loads, truncated extracts — that column-level checks miss entirely.
  • Pipeline idempotency (using Delta MERGE instead of append) is a prerequisite for reliable DQ — validation cannot compensate for non-idempotent ingestion logic.
  • On F2 and F4 SKUs, full-table DQ scans on large Delta tables will cause timeouts — use sampling for non-critical checks and reserve full scans for critical business rules only.
  • Data Activator can monitor your DQ log Delta table and trigger real-time alerts to Teams or email without any custom alerting infrastructure.
