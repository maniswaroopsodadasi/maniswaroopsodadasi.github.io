Why Cross-Workspace Data Sharing Is a Critical Architecture Decision

Imagine your organization has grown organically: the Finance team built their own Fabric workspace with carefully curated Gold-layer Delta tables, the Marketing team has a separate workspace with customer segmentation models, and the Data Platform team maintains a central OneLake-based data lake. Now a new executive dashboard needs data from all three. What do you do — copy everything into a fourth workspace, build fragile pipelines, or find a smarter path?

This scenario plays out in virtually every enterprise Fabric deployment. Cross-workspace data sharing is not a convenience feature — it is a foundational architectural decision that determines whether your Fabric environment scales cleanly or collapses into a tangled web of duplicated datasets, broken refresh chains, and permission nightmares. Getting it right from day one saves months of refactoring later.

In this article, we'll dissect every mechanism Microsoft Fabric provides for sharing data across workspace boundaries, when to use each, and the non-obvious pitfalls that trap even experienced engineers.

Core Concepts: How Fabric Thinks About Workspace Boundaries

The Workspace as a Security and Governance Boundary

In Microsoft Fabric, a workspace is simultaneously a collaboration unit, a billing boundary (tied to a Fabric capacity SKU like F2, F64, or F128), and a security perimeter. Items inside a workspace — Lakehouses, Warehouses, Semantic Models, Notebooks, Pipelines — are governed by workspace-level roles: Admin, Member, Contributor, and Viewer.

The critical insight most beginners miss: workspace roles do not automatically propagate to item-level permissions. A user with Viewer access to Workspace A cannot automatically read a Lakehouse in Workspace B, even if both workspaces sit on the same F64 capacity. Cross-workspace access requires explicit, deliberate configuration — and that configuration choice has deep implications for performance, governance, and maintainability.

OneLake as the Unifying Storage Layer

All Fabric data items — Lakehouses, Warehouses, KQL Databases — store their data in OneLake, Microsoft's single logical data lake built on Azure Data Lake Storage Gen2. OneLake uses a hierarchical namespace structured as onelake://<workspace>/<item>/Tables/<table>. This architecture means that, at the storage layer, data is already centralized. The question is how you expose it across workspace boundaries at the compute and semantic layers.

The Four Primary Mechanisms for Cross-Workspace Data Sharing

1. Workspace Item Sharing (Direct Access)

The simplest mechanism: share a specific Fabric item (e.g., a Lakehouse or Warehouse) with users or groups from another workspace. Navigate to the item, select Share, and grant Read or ReadAll permissions. The recipient can then query the item directly from their own workspace using cross-workspace references.

In a Notebook in Workspace B, you can reference a Lakehouse from Workspace A using its fully qualified OneLake path:

from pyspark.sql import SparkSession

# Reference a Lakehouse in a different workspace using its OneLake ABFS path
# Format: abfss://<workspace_id>@onelake.dfs.fabric.microsoft.com/<lakehouse_id>/Tables/

workspace_id = "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
lakehouse_id = "f9e8d7c6-b5a4-3210-fedc-ba9876543210"

finance_gold_path = f"abfss://{workspace_id}@onelake.dfs.fabric.microsoft.com/{lakehouse_id}/Tables/fact_revenue"

df_revenue = spark.read.format("delta").load(finance_gold_path)
df_revenue.printSchema()
df_revenue.filter("fiscal_year = 2024").show(10)

This approach reads Delta Parquet files directly from OneLake without any data movement. The compute runs in the calling workspace's capacity, but the storage is read from the source workspace's OneLake path. No data duplication occurs.

2. Lakehouse Shortcuts

Shortcuts are the most architecturally elegant cross-workspace sharing mechanism in Fabric. A shortcut is a pointer — a virtual mount point inside one Lakehouse that references data stored in another location, including another Fabric Lakehouse, Azure Data Lake Storage Gen2, Amazon S3, or Google Cloud Storage.

When you create an OneLake shortcut from Workspace B's Lakehouse pointing to a table in Workspace A's Lakehouse, users in Workspace B see the table as if it were local — it appears in the Lakehouse explorer, is queryable via the SQL analytics endpoint, and is accessible to Notebooks and Pipelines. Crucially, no data is copied; reads go directly to the source Delta Parquet files.

Shortcuts respect the source item's permissions. If the source Lakehouse owner revokes access, the shortcut immediately becomes inaccessible — a powerful governance control that data platform teams often overlook when designing their sharing strategy.

3. Semantic Model Sharing (Power BI / DirectLake)

For BI consumers, the sharing unit is typically a Semantic Model (formerly Power BI dataset). A Semantic Model built in Workspace A using DirectLake mode — which reads Delta Parquet files directly from OneLake without import — can be shared with report authors in Workspace B using Build permission. Those authors can then create reports in Workspace B that live-connect to the shared Semantic Model.

This is the recommended pattern for enterprise BI governance: one certified Semantic Model, many downstream reports across many workspaces. The DirectLake connection means the model always reflects the latest data in the underlying Lakehouse without scheduled refreshes.

4. Warehouse Cross-Database Queries (T-SQL Federation)

Fabric Warehouses support cross-database T-SQL queries using three-part naming: [workspace_name].[warehouse_name].[schema].[table]. This allows analysts to write SQL that joins tables from multiple warehouses across workspace boundaries, similar to linked servers in SQL Server but without the configuration overhead.

-- Cross-workspace T-SQL query in Fabric Warehouse
-- Joining Finance warehouse (Workspace A) with Marketing warehouse (Workspace B)
-- Both workspaces must be on the same Fabric capacity or have cross-capacity sharing enabled

SELECT
    f.customer_id,
    f.total_revenue_ytd,
    f.fiscal_quarter,
    m.segment_name,
    m.churn_probability_score,
    m.last_campaign_response_date
FROM
    [FinanceWorkspace].[FinanceWarehouse].[gold].[fact_customer_revenue] AS f
INNER JOIN
    [MarketingWorkspace].[MarketingWarehouse].[gold].[dim_customer_segment] AS m
    ON f.customer_id = m.customer_id
WHERE
    f.fiscal_quarter = 'Q4-2024'
    AND m.churn_probability_score > 0.65
ORDER BY
    f.total_revenue_ytd DESC;

The key constraint: cross-warehouse queries require the querying user to have at least ReadData permission on the referenced warehouse. The query executes in the calling warehouse's compute context, meaning it consumes capacity from the workspace where the query is initiated.

Best Practices for Cross-Workspace Data Sharing

  • Prefer Shortcuts over ABFS path reads in production pipelines. Shortcuts appear as first-class Lakehouse objects, making lineage tracking in Fabric's data catalog accurate. Raw ABFS paths in Notebooks are invisible to the catalog and create undocumented dependencies.
  • Establish a dedicated "Gold" or "Certified Data Products" workspace per domain. Rather than sharing from team workspaces (which change frequently), publish stable, versioned Gold tables to a domain workspace with controlled write access. All downstream consumers shortcut from there.
  • Use Semantic Model Build permissions for BI sharing, never workspace Viewer access. Granting workspace Viewer access to report authors exposes all items in the workspace. Build permission on a specific Semantic Model is surgical and auditable.
  • Apply Row-Level Security (RLS) in shared Semantic Models before sharing. RLS defined in the source Semantic Model is enforced for all downstream report consumers regardless of which workspace the report lives in. This is your primary data access control for BI.
  • Monitor shortcut access patterns using Fabric Capacity Metrics app. Shortcuts that are queried heavily via the SQL analytics endpoint consume CUs (Capacity Units) from the source workspace's capacity, not the consumer's. This surprises teams when their capacity bills spike unexpectedly.
  • Version-control your shortcut topology. Document which workspaces consume shortcuts from which sources. A source Lakehouse rename or restructure silently breaks all downstream shortcuts — there is no built-in dependency graph alert as of early 2025.
  • For cross-capacity sharing, verify network and tenant settings. Cross-workspace sharing across different Fabric capacities (e.g., F64 in East US and F32 in West Europe) requires the tenant admin to enable Allow OneLake cross-region access in the Fabric Admin Portal. Missing this setting causes cryptic permission errors.

Common Pitfalls

Pitfall Cause Fix
Shortcut shows data but SQL endpoint returns "Object not found" Shortcut was created to the Files section of the source Lakehouse, not the Tables section. SQL endpoint only exposes Delta tables in the Tables folder. Recreate the shortcut pointing to /Tables/<table_name> and ensure the source is a registered Delta table, not raw Parquet files.
Cross-warehouse T-SQL query returns stale data The referenced warehouse uses Import-mode Semantic Models with scheduled refresh; the warehouse itself is current but the analyst is querying a cached model layer. Query the warehouse SQL endpoint directly (not through a Semantic Model) for real-time data. Use DirectLake Semantic Models for near-real-time BI.
Shared Semantic Model loses DirectLake connection after workspace move Moving a Lakehouse to a new workspace changes its OneLake path (workspace_id changes). The Semantic Model's DirectLake connection string becomes invalid. After any workspace migration, re-bind the Semantic Model to the new Lakehouse using the XMLA endpoint or Fabric REST API. Automate this in your workspace provisioning scripts.
Capacity CU spikes in source workspace from shortcut reads Consumer workspaces query shortcuts via the SQL analytics endpoint; compute runs against the source workspace's capacity, not the consumer's. Implement query throttling using Fabric's Capacity Burst settings. Consider moving high-volume shared tables to a dedicated shared-services workspace on a larger SKU (F64+).
Build permission granted but report author cannot see Semantic Model in Power BI Desktop The Semantic Model is in a workspace the author has no workspace-level role in. Build permission alone is insufficient for discovery in some client tools. Grant the author at least Viewer role on the source workspace, OR share the Semantic Model link directly. Alternatively, publish the Semantic Model to a shared workspace where the author has Viewer access.
ABFS path read fails with "AuthorizationPermissionMismatch" in Notebook The Notebook runs under the workspace's Managed Identity or the user's identity, which lacks ReadAll permission on the source Lakehouse item. Grant ReadAll (not just Read) on the source Lakehouse item to the calling workspace's identity. Read permission allows metadata access; ReadAll is required for data file access.

Real-World Scenario: Building a Cross-Domain Executive Dashboard

A retail company has three domain workspaces: Finance-Prod (F64 capacity), Supply-Chain-Prod (F32 capacity), and Marketing-Prod (F32 capacity). The CDO wants a unified executive dashboard with revenue, inventory health, and campaign ROI — all in one Power BI report, refreshed in near-real-time, with no data duplication.

  1. Create a "Executive-Analytics" workspace on an F64 capacity. This workspace will own the unified Semantic Model and the executive reports. Assign the BI team as Members and the data platform team as Admins.
  2. Create a new Lakehouse called "exec_gold_lh" inside Executive-Analytics workspace. This Lakehouse will serve as the integration point but will contain no raw data — only shortcuts.
  3. In Finance-Prod workspace, navigate to the Gold Lakehouse → Share → grant ReadAll permission to the Executive-Analytics workspace's service principal (or the BI team's Entra ID group). Do the same for Supply-Chain-Prod and Marketing-Prod.
  4. Inside exec_gold_lh, create three OneLake shortcuts:
    • fact_revenue → pointing to Finance-Prod/FinanceGold_LH/Tables/fact_revenue
    • fact_inventory_health → pointing to Supply-Chain-Prod/SCGold_LH/Tables/fact_inventory_health
    • fact_campaign_roi → pointing to Marketing-Prod/MktGold_LH/Tables/fact_campaign_roi
  5. Verify shortcuts by querying the SQL analytics endpoint of exec_gold_lh in the Fabric SQL editor. All three tables should appear and return data without any pipeline execution.
  6. Create a DirectLake Semantic Model in Executive-Analytics workspace connected to exec_gold_lh. Add all three shortcut tables. Build the data model with relationships on date_key and product_id.
  7. Apply Row-Level Security in the Semantic Model: regional VPs should only see their region's revenue data. Define RLS roles using DAX filter expressions on the dim_region table.
  8. Grant Build permission on the Semantic Model to the BI report authors. They can now create reports in their own team workspaces that connect to this certified Semantic Model.
  9. Publish the executive report to Executive-Analytics workspace and embed it in the executive SharePoint portal. Because DirectLake is used, the report reflects data as soon as the source Delta tables are updated — no refresh schedule needed.
  10. Set up Fabric Capacity Metrics monitoring on the F64 Executive-Analytics capacity. Watch for CU spikes during business hours when the dashboard is heavily used, and configure smoothing windows to prevent throttling.

The result: a fully governed, near-real-time executive dashboard with zero data duplication, enforced RLS, and clear data lineage back to three source domain workspaces — all achieved without a single pipeline copy activity.

Key Takeaways

  • OneLake Shortcuts are the preferred cross-workspace sharing mechanism for data engineering use cases — they create zero data duplication, appear as first-class Lakehouse objects, and propagate source permissions automatically.
  • Semantic Model Build permissions are the correct BI sharing primitive — never grant workspace Viewer access just to enable report authoring against a specific dataset.
  • Compute costs follow the query, not the data. Shortcut reads via SQL analytics endpoint consume CUs from the source workspace's capacity. Design your capacity topology with this in mind, especially for high-volume shared tables.
  • Cross-workspace T-SQL federation in Fabric Warehouse enables powerful ad-hoc analytics without ETL, but adds query planning overhead and requires explicit ReadData grants on each referenced warehouse.
  • DirectLake Semantic Models break silently when their source Lakehouse moves. Automate re-binding as part of any workspace migration runbook.
  • A dedicated shared-services or domain Gold workspace is the most scalable pattern for large organizations — it decouples source domain teams from consumer teams and provides a stable, versioned data contract.
  • The tenant-level "Allow OneLake cross-region access" setting is a hidden prerequisite for cross-capacity sharing across Azure regions — verify this before designing multi-region Fabric architectures.
