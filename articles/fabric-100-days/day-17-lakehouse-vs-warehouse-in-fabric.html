Lakehouse vs Warehouse in Fabric

Why This Decision Can Make or Break Your Fabric Implementation

One of the most consequential architectural decisions you will make in Microsoft Fabric is choosing between a Lakehouse and a Warehouse as your primary analytical store. Get it wrong, and you will spend months fighting the platform instead of delivering value. Get it right, and your entire data stack snaps together with surprising elegance.

This is not a theoretical debate. Real teams are making this choice today, often under time pressure, and many are defaulting to the Warehouse simply because it feels familiar — it looks like SQL Server, it has T-SQL, and their analysts already know it. Others are defaulting to the Lakehouse because it sounds modern and "data lake-ish." Both instincts can lead to the wrong outcome.

By the end of this article, you will understand the architectural DNA of each option, where each genuinely excels, and how to make a defensible, context-aware decision for your specific workload.

Core Concepts: What Are These Things, Really?

The Fabric Lakehouse

A Fabric Lakehouse is a unified storage and compute experience built on top of OneLake, Microsoft Fabric's single logical data lake. Every table in a Lakehouse is stored as Delta Parquet files in OneLake — there is no proprietary format, no vendor lock-in at the storage layer. The Lakehouse exposes two endpoints simultaneously: a SQL Analytics Endpoint (read-only T-SQL access) and a Spark compute engine for full read/write operations using PySpark, Spark SQL, or Scala.

The critical non-obvious insight here is that the SQL Analytics Endpoint is automatically generated — you do not create it, configure it, or maintain it. Every Delta table you write via Spark, Dataflow Gen2, or a pipeline automatically appears in the SQL Analytics Endpoint within seconds. This is powered by the Delta Lake transaction log, which Fabric reads to keep the metadata in sync.

The Fabric Warehouse

A Fabric Warehouse is a fully managed, cloud-native data warehouse with a dedicated SQL engine that supports full read/write T-SQL, including DDL statements like CREATE TABLE, ALTER TABLE, and DML like INSERT, UPDATE, and DELETE. Under the hood, Warehouse data is also stored in OneLake as Delta Parquet — this is a crucial architectural fact that most people miss. The Warehouse is not a separate silo; it is a different compute and governance layer over the same OneLake storage.

The Warehouse provides a traditional relational experience: you define schemas explicitly, enforce constraints (informational, not enforced at write time), write stored procedures, and manage table statistics. It is the right tool when your team lives in SQL and needs full transactional write semantics through a SQL interface.

The Architectural Difference That Actually Matters

Both stores use Delta Parquet on OneLake. Both are queryable via T-SQL. Both support DirectLake mode in Power BI semantic models. So what is actually different?

The difference is who writes the data and how:

  • Lakehouse: Data is written primarily by Spark (notebooks, Spark job definitions), Dataflow Gen2, or external tools that write Delta files directly to OneLake. The SQL Analytics Endpoint is read-only — you cannot run INSERT INTO or UPDATE statements against it.
  • Warehouse: Data is written exclusively through the SQL engine using T-SQL DML. Spark cannot write directly to Warehouse-managed tables (though you can read them via shortcuts).
  • Schema control: In a Lakehouse, schema is inferred from the Delta log and can be schema-on-read. In a Warehouse, schema is explicitly defined and enforced by the SQL engine.
  • Cross-database queries: Both support cross-item queries using three-part naming (workspace.item.schema.table), but the Warehouse has more mature support for cross-warehouse joins with full predicate pushdown.
  • Compute isolation: Lakehouse Spark compute is billed separately from SQL compute. Warehouse SQL compute draws from your Fabric capacity (F-SKU) directly. On an F2 SKU, running heavy Spark jobs alongside Warehouse queries can cause throttling — a real operational concern.

When to Choose Each: Decision Framework

Choose a Lakehouse When

  • Your ingestion pipeline is Spark-based — notebooks, Spark job definitions, or Dataflow Gen2 writing Delta tables. Fighting the Lakehouse's read-only SQL endpoint is the wrong battle.
  • You need to store raw, unstructured, or semi-structured files (JSON, CSV, Parquet, images) alongside your Delta tables. The Lakehouse Files section is a first-class OneLake folder — the Warehouse has no equivalent.
  • You are building a medallion architecture (Bronze → Silver → Gold) where each layer is transformed by Spark. The Lakehouse is the natural home for all three layers.
  • You need Delta time travel for auditing or reprocessing — querying a table as it existed 30 days ago using FOR TIMESTAMP AS OF in Spark SQL.
  • Your team includes data scientists who need direct access to raw Delta files for model training without going through a SQL interface.
  • You are on a lower F-SKU (F2, F4) and need to minimize SQL compute consumption — Spark jobs run on separate Spark pools that do not consume your Fabric capacity in the same way.

Choose a Warehouse When

  • Your transformation logic is written in T-SQL and your team has deep SQL expertise but limited Spark experience. Stored procedures, CTEs, and window functions work exactly as expected.
  • You need full DML operations — UPDATE, DELETE, MERGE — executed through SQL without spinning up a Spark session. Spark session startup latency (30–90 seconds on a cold pool) is unacceptable for interactive workflows.
  • You are migrating from Azure Synapse Analytics dedicated SQL pools or Azure SQL Data Warehouse and want a familiar operational model with minimal retraining.
  • You need robust row-level security defined and managed entirely in T-SQL, applied consistently across all BI tools connecting via the SQL connection string.
  • Your Gold layer is consumed primarily by Power BI reports and the semantic model team needs a stable, explicitly-typed schema that does not shift when upstream Spark jobs change column types.
  • You need cross-database joins across multiple Warehouses with reliable performance — the Warehouse optimizer handles distributed joins more predictably than the SQL Analytics Endpoint.

Best Practices

Do Avoid
Use a Lakehouse for Bronze and Silver layers in a medallion architecture, then expose Gold via a Warehouse or a dedicated Lakehouse SQL endpoint Avoid writing your Gold/serving layer directly to a Lakehouse if your BI team needs to run ad-hoc UPDATE statements for corrections
Use OneLake shortcuts to give your Warehouse read access to Lakehouse Delta tables without data duplication Avoid copying data from a Lakehouse into a Warehouse using pipelines — this creates two copies of the same data in OneLake and doubles storage costs
Define explicit column types and table statistics in your Warehouse Gold layer to ensure consistent DirectLake semantic model behavior Avoid relying on schema inference in the SQL Analytics Endpoint for production Power BI reports — inferred types can change if upstream Spark jobs change
Use the Lakehouse Files section to land raw source files before processing them into Delta tables with Spark Avoid storing raw files in a Warehouse — it has no Files section and is not designed for unstructured data
Enable V-Order optimization on Delta tables in your Lakehouse Gold layer to accelerate DirectLake reads in Power BI Avoid disabling V-Order in Spark write operations if those tables will be consumed by Power BI DirectLake — query performance degrades measurably
Use Warehouse stored procedures for complex multi-step SQL transformations that need transactional consistency Avoid using the Lakehouse SQL Analytics Endpoint for transformation logic — it is read-only and any workaround using CETAS (CREATE EXTERNAL TABLE AS SELECT) is not supported in Fabric's SQL endpoint

Real-World Scenario: Retail Analytics Platform

A mid-size retailer needs to build an analytics platform in Fabric to serve three audiences: data engineers processing 50 million daily transaction records, a data science team building demand forecasting models, and a BI team delivering executive dashboards in Power BI. Here is how the architecture decision plays out step by step:

  1. Define the medallion layers: The team decides on three Lakehouses — Bronze_LH, Silver_LH, and Gold_LH — plus one Warehouse — Serving_WH. This separation of concerns is the foundation of the entire design.
  2. Land raw data in Bronze_LH: A Data Pipeline copies raw CSV transaction files from Azure Data Lake Storage Gen2 into the Files section of Bronze_LH. No Spark processing yet — just raw files preserved for reprocessing. This is only possible in a Lakehouse.
  3. Transform Bronze to Silver using Spark: A Spark notebook reads the raw CSVs, applies data quality rules, and writes cleaned Delta tables to Silver_LH. The Spark job runs on a Medium Spark pool (8 vCores, 56 GB RAM). The SQL Analytics Endpoint on Silver_LH immediately reflects the new tables.
  4. Build Gold aggregations in Gold_LH: Another Spark notebook joins Silver_LH tables to produce daily sales aggregates, customer segments, and product performance metrics as Delta tables in Gold_LH. V-Order optimization is enabled on all writes to prepare for DirectLake.
  5. Expose the serving layer via Serving_WH: The BI team needs to add manual corrections to certain dimension records and run complex multi-step SQL transformations. The team creates OneLake shortcuts in Serving_WH pointing to the Gold_LH Delta tables — no data is copied. Additional enrichment tables are created natively in Serving_WH using T-SQL DDL.
  6. Apply row-level security in Serving_WH: The BI team defines T-SQL RLS policies in Serving_WH so that regional managers only see their own store data. This is enforced at the SQL engine level and applies to all Power BI reports connecting via the Warehouse SQL connection string.
  7. Build Power BI semantic model in DirectLake mode: The Power BI team creates a semantic model pointing to Serving_WH. Because the Gold tables are exposed as shortcuts with V-Order-optimized Delta files, DirectLake mode is active — sub-second query performance on 500 million rows without import mode data refresh.
  8. Data science team accesses Silver_LH directly: The ML team reads Silver_LH Delta tables directly from Spark notebooks for feature engineering. They use Delta time travel to access last month's snapshot for model validation. They never touch the Warehouse — their workflow is entirely Spark-native.

The result: each team works in their natural environment, no data is duplicated in OneLake, and the platform runs efficiently on an F32 SKU with clear workload separation.

Key Takeaways

  • Both Lakehouse and Warehouse store data as Delta Parquet on OneLake — the difference is the compute engine and write semantics, not the storage layer.
  • The Lakehouse SQL Analytics Endpoint is read-only — this is the single most important fact to internalize before designing your architecture.
  • Use OneLake shortcuts to give a Warehouse read access to Lakehouse tables without copying data — this is the correct integration pattern, not pipelines.
  • Lakehouses are the natural home for medallion architecture Bronze and Silver layers; Warehouses shine as the serving/Gold layer when SQL teams need write access.
  • Both support DirectLake mode in Power BI, but Warehouse provides more stable schema guarantees for production semantic models.
  • V-Order optimization on Delta writes in the Lakehouse directly impacts Power BI DirectLake query performance — do not skip it.
  • On lower F-SKUs (F2, F4), be deliberate about workload scheduling — Spark and Warehouse SQL queries compete for the same capacity units.
  • Row-level security is more mature and operationally simpler in the Warehouse — if RLS is a hard requirement, weight your Gold layer toward the Warehouse.
