Why Streaming Data Matters: The Cost of Stale Insights

Imagine you run fraud detection for a financial services company. Your batch ETL pipeline refreshes every four hours. A fraudulent transaction pattern emerges at 9:02 AM. Your analysts don't see it until 1:00 PM — nearly four hours later, after hundreds of additional fraudulent transactions have cleared. This is the core business problem that streaming data architectures solve: the gap between when something happens and when your systems know about it.

Microsoft Fabric's streaming capabilities close this gap by providing a unified platform where data can be ingested, processed, and made available for analytics within seconds to minutes — not hours. On Day 28 of our series, we move beyond batch processing into the real-time world, covering Eventstream, KQL Databases, Real-Time Intelligence, and how these components integrate with the broader Fabric ecosystem including OneLake and Delta Parquet storage.

Core Concepts in Microsoft Fabric Streaming

The Real-Time Intelligence Workload

Microsoft Fabric organizes streaming capabilities under the Real-Time Intelligence workload (formerly called Real-Time Analytics). This workload contains three primary artifacts you'll work with daily:

  • Eventstream — A no-code/low-code pipeline builder for ingesting, transforming, and routing streaming data from sources like Azure Event Hubs, Azure IoT Hub, Kafka, and custom endpoints. Think of it as Stream Analytics but deeply integrated into Fabric's unified compute and storage model.
  • KQL Database (Kusto Query Language Database) — A columnar, time-series-optimized analytical store purpose-built for high-velocity append-only data. It supports sub-second query latency on billions of rows and is the primary destination for streaming data in Fabric.
  • KQL Queryset — A workbook-style interface for writing, saving, and sharing KQL queries against one or more KQL Databases. Equivalent to a SQL query editor but for Kusto.

How Streaming Data Flows Through Fabric

A typical streaming architecture in Fabric follows this path:

  1. An event source (IoT device, application log, clickstream) publishes messages to Azure Event Hubs or a Kafka topic.
  2. Eventstream connects to that source and applies optional transformations — filtering, aggregation, schema mapping — using a visual canvas.
  3. Eventstream routes the processed data to one or more destinations: a KQL Database for real-time querying, a Lakehouse Delta table for batch analytics, or another Eventstream for fan-out patterns.
  4. Analysts query the KQL Database using KQL Querysets, Power BI Real-Time dashboards, or Reflex alerts.
  5. Optionally, KQL Database shortcuts or OneLake mirroring make the data available in Delta Parquet format for downstream Spark notebooks or SQL analytics endpoints.

This architecture is non-obvious in one critical way: the KQL Database and the Lakehouse are not the same store. Data written to a KQL Database lives in Kusto's proprietary columnar format, not in Delta Parquet. If you need the same streaming data available in both real-time (sub-second KQL queries) and batch (Spark/SQL), you must explicitly route to both destinations from Eventstream, or use OneLake availability features introduced in Fabric's 2024 updates.

Transformations in Eventstream

Eventstream's transformation canvas supports operations that execute as continuous streaming queries. Key operators include:

  • Filter — Drop events that don't match a predicate (e.g., temperature > 80). Reduces downstream storage costs significantly when only a fraction of events are analytically relevant.
  • Aggregate — Tumbling, hopping, and session windows. A tumbling window of 30 seconds computes metrics like average CPU utilization per device every 30 seconds, emitting one row per window per group key.
  • Group By — Partition aggregations by one or more fields (e.g., device ID, region).
  • Expand — Flatten nested JSON arrays into individual rows. Critical for IoT payloads where a single message contains an array of sensor readings.
  • Union — Merge two Eventstream branches into a single output stream.
  • Manage Fields — Add, rename, or drop columns. Use this to normalize schema differences between upstream producers before writing to a KQL table with a fixed schema.

KQL Database: Architecture and Capabilities

Why KQL Instead of SQL for Streaming?

KQL Databases use a columnar storage engine (derived from Azure Data Explorer) that is architecturally optimized for time-series and log data. Several characteristics make it superior to a traditional SQL engine for streaming workloads:

  • Ingestion latency of 5–10 seconds from event source to queryable data, compared to Delta table streaming which typically achieves 30–60 second micro-batch latency.
  • Automatic data sharding — Kusto automatically manages data extents (shards) and merges them in the background. You don't manually tune partition strategies for time-series data.
  • Native time-series functions — Functions like make-series, series_decompose_anomalies(), and series_fill_linear() are first-class KQL operators with no equivalent in Spark SQL without custom UDFs.
  • Hot and cold cache tiers — You can configure a KQL Database to keep the last 7 days in hot (SSD) cache and 90 days in cold storage, dramatically reducing query costs on recent data while retaining historical data affordably.

Writing KQL Queries for Streaming Data

KQL has a pipe-based syntax that reads left to right, making it highly readable for complex analytical queries. Here is a realistic KQL query for analyzing IoT sensor anomalies in a manufacturing scenario:

// Detect temperature anomalies using a 5-minute tumbling window
// Table: SensorReadings | Columns: DeviceId, SensorType, Value, EventTimestamp
SensorReadings
| where EventTimestamp >= ago(1h)
| where SensorType == "temperature"
| summarize
    AvgTemp = avg(Value),
    MaxTemp = max(Value),
    ReadingCount = count()
    by DeviceId, bin(EventTimestamp, 5m)
| where MaxTemp > 95.0 or AvgTemp > 85.0
| extend AnomalyFlag = iff(MaxTemp > 95.0, "CRITICAL", "WARNING")
| order by EventTimestamp desc, MaxTemp desc
| project EventTimestamp, DeviceId, AvgTemp = round(AvgTemp, 2), MaxTemp, ReadingCount, AnomalyFlag

This query is non-trivial: the bin() function creates time buckets, summarize aggregates within each bucket per device, and iff() applies conditional logic inline. The entire query typically executes in under 500ms against 100 million rows on an F64 SKU with hot cache enabled.

Ingesting Streaming Data with Python and the Kusto SDK

For programmatic ingestion — useful when you need to stream from a custom source not natively supported by Eventstream — you can use the azure-kusto-data and azure-kusto-ingest Python libraries:

from azure.kusto.data import KustoClient, KustoConnectionStringBuilder
from azure.kusto.ingest import QueuedIngestClient, IngestionProperties, DataFormat
import pandas as pd
from datetime import datetime, timezone

# --- Connection Configuration ---
CLUSTER_URI = "https://your-fabric-kql-cluster.kusto.fabric.microsoft.com"
DATABASE = "SensorAnalytics"
TABLE = "SensorReadings"

# Use managed identity in production; interactive auth for development
kcsb = KustoConnectionStringBuilder.with_interactive_login(CLUSTER_URI)

# QueuedIngestClient provides reliable, at-least-once delivery
ingest_client = QueuedIngestClient(
    KustoConnectionStringBuilder.with_interactive_login(
        f"https://ingest-{CLUSTER_URI.replace('https://', '')}"
    )
)

# --- Simulate a batch of sensor readings ---
readings = pd.DataFrame({
    "DeviceId": ["device-001", "device-002", "device-001"],
    "SensorType": ["temperature", "temperature", "humidity"],
    "Value": [87.3, 96.1, 62.4],
    "EventTimestamp": [
        datetime.now(timezone.utc),
        datetime.now(timezone.utc),
        datetime.now(timezone.utc)
    ]
})

# --- Ingest the DataFrame ---
ingestion_props = IngestionProperties(
    database=DATABASE,
    table=TABLE,
    data_format=DataFormat.CSV,
    # Mapping reference must match a pre-created ingestion mapping in KQL DB
    ingestion_mapping_reference="SensorReadings_CSV_Mapping"
)

from azure.kusto.ingest import FileDescriptor
import io

csv_buffer = io.StringIO()
readings.to_csv(csv_buffer, index=False, header=False)
csv_buffer.seek(0)

# For production: use BlobDescriptor for large payloads > 1 MB
ingest_client.ingest_from_stream(
    csv_buffer,
    ingestion_properties=ingestion_props
)

print(f"Queued {len(readings)} records for ingestion into {DATABASE}.{TABLE}")
# Note: Queued ingestion has ~5-10 minute latency; use streaming ingestion
# policy on the table for sub-second latency on smaller payloads

A critical non-obvious detail: QueuedIngestClient routes data through Azure Blob Storage queues before loading into the KQL engine. This provides durability but adds 5–10 minutes of latency. If you need sub-second ingestion latency programmatically, enable the streaming ingestion policy on the target table with .alter table SensorReadings policy streamingingestion enable and use StreamingIngestClient instead.

Best Practices for Streaming Data in Fabric

Do Avoid
Enable streaming ingestion policy on KQL tables receiving <1 MB/s to achieve sub-10-second latency Using queued ingestion for latency-sensitive use cases — it adds 5–10 minutes of delay
Route to both KQL DB and Lakehouse from Eventstream when you need real-time dashboards AND historical batch analysis Assuming KQL DB data is automatically available as Delta Parquet in OneLake — it is not by default
Use the bin() function in KQL to pre-aggregate time buckets before Power BI queries to reduce CU consumption Running raw event-level queries from Power BI against KQL DB — this causes full table scans on every dashboard refresh
Set table-level retention policies (e.g., 30 days hot, 365 days total) to control storage costs proactively Accepting the default 365-day hot cache for all tables — this dramatically increases CU costs for rarely queried historical data
Use Eventstream's Filter node to drop irrelevant events before they reach the KQL DB Ingesting all raw events and filtering at query time — storage and ingestion CUs are consumed regardless of query patterns
Monitor Eventstream health using the built-in Data Preview and Metrics panels to detect backpressure early Ignoring Eventstream operational metrics until data freshness SLAs are breached in production

Common Pitfalls

Pitfall Cause Fix
KQL table schema mismatch causes silent data loss Eventstream sends a field not present in the KQL table schema; Kusto drops the column silently by default Create an ingestion mapping in KQL DB that explicitly maps source fields to table columns; enable .show ingestion failures monitoring
Eventstream stops processing after Event Hubs consumer group conflict Two Eventstream instances share the same consumer group on the same Event Hub, causing partition lease conflicts Create a dedicated consumer group per Eventstream instance in Azure Event Hubs; never share $Default in production
Power BI Real-Time dashboard shows stale data despite streaming ingestion Dashboard auto-refresh is set to 30 minutes (default) instead of the minimum 1-second refresh for streaming tiles Edit each tile's refresh rate in the dashboard settings; streaming visuals require the "Streaming dataset" tile type, not standard KQL-backed visuals
Spark Structured Streaming job fails with "Delta log checkpoint corruption" Two concurrent Spark streaming jobs write to the same Delta table path in OneLake without coordination Use a single writer per Delta table; implement Delta's concurrent write protocol with optimistic concurrency only for batch writers, not streaming
Eventstream aggregate node produces incorrect counts Using a hopping window with hop size larger than window size, which is mathematically invalid Ensure hop size ≤ window size for hopping windows; use tumbling windows (hop = window) when each event should appear in exactly one bucket
KQL Database CU consumption spikes unexpectedly Materialized views not created for frequently run aggregation queries; every query recomputes from raw events Create KQL materialized views with .create materialized-view for aggregations run more than once per minute; views are incrementally maintained by the engine

Real-World Scenario: Building a Real-Time Fleet Monitoring System

A logistics company operates 2,000 delivery vehicles, each emitting GPS coordinates, engine temperature, fuel level, and speed every 10 seconds via Azure IoT Hub. The operations team needs a live dashboard showing vehicles exceeding speed limits and engine temperature alerts within 15 seconds of the event occurring. Here is the step-by-step implementation:

  1. Provision the Fabric capacity — Use an F32 SKU to support the expected ingestion rate of approximately 2,000 events × 10 seconds = 200 events/second, with headroom for query concurrency. F16 would be borderline; F32 provides comfortable margin.
  2. Create a KQL Database — In the Fabric workspace, create a new KQL Database named FleetMonitoring. Run the following KQL to create the target table and enable streaming ingestion: 
    .create table VehicleTelemetry (
    VehicleId: string,
    EventTimestamp: datetime,
    Latitude: real,
    Longitude: real,
    SpeedKmh: real,
    EngineTemp: real,
    FuelLevelPct: real
)

.alter table VehicleTelemetry policy streamingingestion enable

.create table VehicleTelemetry ingestion csv mapping 'VehicleTelemetry_Mapping'
'[{"Column":"VehicleId","Properties":{"Ordinal":0}},
  {"Column":"EventTimestamp","Properties":{"Ordinal":1}},
  {"Column":"Latitude","Properties":{"Ordinal":2}},
  {"Column":"Longitude","Properties":{"Ordinal":3}},
  {"Column":"SpeedKmh","Properties":{"Ordinal":4}},
  {"Column":"EngineTemp","Properties":{"Ordinal":5}},
  {"Column":"FuelLevelPct","Properties":{"Ordinal":6}}]'
  3. Create a dedicated IoT Hub consumer group — In the Azure Portal, navigate to the IoT Hub → Built-in endpoints → Add consumer group named fabric-eventstream-fleet. Never use $Default.
  4. Build the Eventstream — Create a new Eventstream named FleetTelemetryStream. Add Azure IoT Hub as the source, selecting the fabric-eventstream-fleet consumer group. Add a Filter node to pass only events where SpeedKmh > 0 (filtering out parked vehicles reduces volume by ~40% during off-hours). Add a Manage Fields node to drop internal IoT Hub metadata fields not needed in the KQL table. Connect the output to the FleetMonitoring KQL Database, targeting the VehicleTelemetry table.
  5. Create a KQL materialized view for alerts — Rather than querying raw events in the dashboard, create a materialized view that pre-computes the last known state per vehicle: 
    .create materialized-view with (backfill=true) VehicleLatestState on table VehicleTelemetry
{
    VehicleTelemetry
    | summarize
        arg_max(EventTimestamp, *) // Keep the row with the latest timestamp per vehicle
        by VehicleId
}
  6. Build the alert query — Create a KQL Queryset with the following query that the Power BI dashboard will use: 
    VehicleLatestState
| where EventTimestamp >= ago(2m) // Only vehicles reporting in last 2 minutes (active)
| where SpeedKmh > 120 or EngineTemp > 105
| extend AlertType = case(
    EngineTemp > 105 and SpeedKmh > 120, "CRITICAL: Speed + Overheating",
    EngineTemp > 105, "WARNING: Engine Overheating",
    SpeedKmh > 120, "WARNING: Speed Violation",
    "OK"
  )
| project VehicleId, EventTimestamp, SpeedKmh, EngineTemp, FuelLevelPct, AlertType, Latitude, Longitude
| order by EventTimestamp desc
  7. Connect Power BI — Create a Power BI report connected to the KQL Database using DirectQuery mode. Set the page auto-refresh to 10 seconds (requires Power BI Premium / Fabric F SKU). The materialized view ensures the query returns in under 200ms even at full scale.
  8. Configure Reflex alerts — Create a Data Activator Reflex on the VehicleLatestState materialized view to send Teams notifications when EngineTemp > 110, providing push-based alerting without polling overhead.

Key Takeaways

  • Microsoft Fabric's Real-Time Intelligence workload (Eventstream + KQL Database) achieves 5–30 second end-to-end latency from event source to queryable data — significantly faster than Delta table micro-batch streaming.
  • KQL Databases store data in Kusto's proprietary columnar format, not Delta Parquet. If you need the same data available for Spark or SQL analytics, you must explicitly route to a Lakehouse from Eventstream or configure OneLake availability.
  • Always create dedicated Event Hubs consumer groups per Eventstream instance. Sharing consumer groups causes partition lease conflicts that silently halt data ingestion.
  • Enable streaming ingestion policy on KQL tables for sub-10-second latency; queued ingestion (the default) adds 5–10 minutes of delay that is unacceptable for real-time use cases.
  • KQL materialized views are the single most impactful optimization for streaming dashboards — they shift aggregation cost from query time to ingestion time, enabling sub-200ms dashboard queries on billions of rows.
  • The F SKU tier directly impacts streaming throughput: F2 handles ~5 MB/s ingress on custom endpoints, while F64 handles ~40 MB/s via Event Hubs. Size your capacity based on peak event volume, not average.
  • Use Eventstream's Filter and Manage Fields nodes aggressively before data reaches the KQL Database — every byte ingested consumes CU capacity and storage, even if never queried.
