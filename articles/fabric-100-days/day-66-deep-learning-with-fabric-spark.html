Deep Learning with Fabric Spark: Tackling Unstructured Data at Scale

In today's data-driven world, businesses are awash with information that goes far beyond structured rows and columns. Think about the deluge of customer reviews, social media posts, medical images, sensor data, and video feeds. Extracting meaningful insights from this unstructured and semi-structured data is where traditional machine learning often hits its limits. This is precisely where Deep Learning (DL) shines, offering unparalleled capabilities in pattern recognition, feature extraction, and prediction from complex data types.

However, training deep neural networks, especially large-scale models like those used in Natural Language Processing (NLP) or computer vision, demands significant computational resources and efficient data handling. Datasets can span terabytes, and models can have billions of parameters. Running these workloads on a single machine is often impractical or impossible. This is the "why it matters" for Fabric Spark: it provides the distributed computing backbone necessary to scale deep learning training and inference across vast datasets, seamlessly integrating with your existing data estate in OneLake. For data engineers, analysts, and BI professionals venturing into AI, understanding how to leverage Fabric Spark for deep learning opens up a new realm of possibilities for solving complex business problems, from enhancing customer experience through sentiment analysis to optimizing operations with predictive maintenance on sensor data.

Core Concepts: Deep Learning, Spark, and Fabric's Synergy

What is Deep Learning?

Deep Learning is a subfield of machine learning inspired by the structure and function of the human brain. It uses artificial neural networks with multiple layers (hence "deep") to learn representations of data with multiple levels of abstraction. Unlike traditional ML algorithms that often require manual feature engineering, deep learning models can automatically discover intricate features from raw data. Key architectures include:

  • Convolutional Neural Networks (CNNs): Primarily used for image and video analysis.
  • Recurrent Neural Networks (RNNs) & LSTMs/GRUs: Excellent for sequential data like time series and natural language.
  • Transformers: The state-of-the-art for NLP tasks, enabling powerful models like BERT, GPT, and T5.

These models excel in tasks such as image classification, object detection, speech recognition, machine translation, and text generation, all of which are increasingly critical for modern businesses.

Why Spark for Distributed Deep Learning?

Apache Spark is a unified analytics engine for large-scale data processing. Its in-memory computation capabilities and distributed architecture make it an ideal platform for deep learning workloads for several reasons:

  • Data Preprocessing at Scale: Deep learning models often require extensive data cleaning, transformation, and feature engineering. Spark's DataFrame API allows for highly scalable and efficient preprocessing of massive datasets, preparing them for model training.
  • Distributed Training: While deep learning frameworks like TensorFlow and PyTorch offer their own distributed training capabilities, integrating them with Spark allows for seamless data distribution and orchestration across a cluster. This is crucial when the dataset is too large to fit into the memory of a single machine or when training time needs to be significantly reduced.
  • Unified Platform: Spark allows you to manage your entire ML pipeline – from data ingestion and preparation to model training, evaluation, and inference – within a single, consistent environment. This simplifies MLOps and reduces the overhead of integrating disparate systems.
  • Fault Tolerance: Spark's resilient distributed dataset (RDD) and DataFrame abstractions provide built-in fault tolerance, ensuring that deep learning jobs can recover from failures without losing progress.

Fabric Spark: The Deep Learning Accelerator

Microsoft Fabric's Spark environment takes these advantages further by providing a fully managed, optimized, and integrated platform for deep learning. Here's how:

  • Pre-installed Libraries: Fabric Spark runtimes come pre-configured with popular deep learning frameworks like TensorFlow, PyTorch, and Keras, along with essential libraries like Hugging Face Transformers and SynapseML (formerly MMLSpark). This significantly reduces setup time and dependency management.
  • SynapseML for Distributed Training: SynapseML is a powerful library that simplifies distributed deep learning on Spark. It provides high-level APIs for training TensorFlow and PyTorch models across Spark executors, handling the complexities of data partitioning, model synchronization, and distributed optimizers.
  • Seamless OneLake Integration: Fabric Spark natively integrates with OneLake, Microsoft Fabric's unified data lake. This means your large-scale unstructured data, often stored in Delta Lake format, is directly accessible to your Spark deep learning jobs without complex data movement, ensuring optimal I/O performance.
  • MLflow Integration: Fabric notebooks and Spark job definitions have built-in MLflow integration, allowing you to track experiments, log metrics, parameters, and models, and manage the lifecycle of your deep learning models effectively.
  • Scalable Compute: Fabric Spark capacities (F-SKUs) provide flexible and scalable compute resources, allowing you to provision the right amount of vCores and memory for your deep learning workloads, from small experiments to large-scale distributed training.

Fabric Spark Environment and Libraries

Fabric Spark notebooks and Spark job definitions run on a managed Apache Spark runtime. As of late 2023/early 2024, Fabric typically defaults to Apache Spark 3.3 or 3.4 with Python 3.10. Crucially, these runtimes come with a rich set of pre-installed libraries essential for deep learning:

  • TensorFlow & Keras: For building and training neural networks.
  • PyTorch: Another leading deep learning framework.

While many libraries are pre-installed, you can also install custom libraries using the workspace library management feature. This allows you to upload wheel files or specify Conda environment files to install specific versions or niche libraries required for your deep learning projects.

Distributed Deep Learning with SynapseML

SynapseML simplifies distributed training for TensorFlow and PyTorch models. It abstracts away much of the complexity of managing distributed processes and data synchronization.

Distributed TensorFlow with SynapseML

The DistributedTensorFlowEstimator allows you to train TensorFlow models across your Spark cluster. You provide a Python function that defines your TensorFlow model and training logic, and SynapseML handles the distribution.

from synapse.ml.deeplearning import DistributedTensorFlowEstimator
from pyspark.ml.feature import VectorAssembler
import tensorflow as tf

# Assume 'processed_df' from previous step has 'text_length' and 'label' columns.
# For a real scenario, 'text_length' would be replaced by actual embeddings or token IDs.

# Prepare features for TensorFlow. TensorFlow models typically expect a single input vector.
# This is a simplification; real DL models take more complex inputs.
assembler = VectorAssembler(inputCols=["text_length"], outputCol="features")
vector_df = assembler.transform(processed_df)

# Define the TensorFlow model and training function
def build_and_train_model():
    # Define a simple Keras model
    model = tf.keras.Sequential([
        tf.keras.layers.Input(shape=(1,)), # Input shape matches 'features' column
        tf.keras.layers.Dense(10, activation='relu'),
        tf.keras.layers.Dense(2, activation='softmax') # 2 classes for binary classification
    ])
    model.compile(optimizer='adam',
                  loss='sparse_categorical_crossentropy',
                  metrics=['accuracy'])
    return model

# Create the DistributedTensorFlowEstimator
# num_workers: Number of Spark executors to use for training.
# model_definition: The function that defines your Keras model.
# input_col: The column containing features for the model.
# label_col: The column containing labels.
# epoch: Number of training epochs.
tf_estimator = DistributedTensorFlowEstimator(
    num_workers=2, # Use 2 Spark executors for training
    model_definition=build_and_train_model,
    input_col="features",
    label_col="label",
    epoch=3,
    batch_size=32,
    verbose=1 # Show training progress
)

# Train the model
tf_model = tf_estimator.fit(vector_df)

Distributed PyTorch with SynapseML

Similarly, DistributedPyTorchEstimator enables distributed PyTorch training. You provide a Python function that returns your PyTorch model, optimizer, and loss function, along with a training loop.

from synapse.ml.deeplearning import DistributedPyTorchEstimator
from pyspark.ml.feature import VectorAssembler
import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import Dataset, DataLoader

# Assume 'processed_df' from previous step has 'text_length' and 'label' columns.
assembler = VectorAssembler(inputCols=["text_length"], outputCol="features")
vector_df = assembler.transform(processed_df)

# Define a custom PyTorch Dataset for Spark DataFrames
class SparkDataset(Dataset):
    def __init__(self, data_iterator):
        self.features = []
        self.labels = []
        for row in data_iterator:
            # Convert Spark Vector to list, then to tensor
            self.features.append(torch.tensor(row.features.toArray(), dtype=torch.float32))
            self.labels.append(torch.tensor(row.label, dtype=torch.long))

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        return self.features[idx], self.labels[idx]

# Define the PyTorch model
class SimpleNN(nn.Module):
    def __init__(self):
        super(SimpleNN, self).__init__()
        self.fc1 = nn.Linear(1, 10) # Input size 1 (text_length)
        self.relu = nn.ReLU()
        self.fc2 = nn.Linear(10, 2) # Output size 2 (for 2 classes)

    def forward(self, x):
        x = self.fc1(x)
        x = self.relu(x)
        x = self.fc2(x)
        return x

# Define the training function that will run on each worker
def train_model(data_iterator, context):
    model = SimpleNN()
    optimizer = optim.Adam(model.parameters(), lr=0.001)
    criterion = nn.CrossEntropyLoss()

    dataset = SparkDataset(data_iterator)
    dataloader = DataLoader(dataset, batch_size=32, shuffle=True)

    for epoch in range(3): # Number of epochs
        for inputs, labels in dataloader:
            optimizer.zero_grad()
            outputs = model(inputs)
            loss = criterion(outputs, labels)
            loss.backward()
            optimizer.step()
        print(f"Epoch {epoch+1} Loss: {loss.item()}")
    return model.state_dict() # Return model state dictionary

# Create the DistributedPyTorchEstimator
pytorch_estimator = DistributedPyTorchEstimator(
    num_workers=2,
    model_definition=train_model,
    input_col="features",
    label_col="label", # Not directly used by model_definition, but required by estimator
    batch_size=32,
    verbose=1
)

# Train the model
pytorch_model = pytorch_estimator.fit(vector_df)

# The trained model (state_dict) can be loaded into a PyTorch model for inference
# loaded_model = SimpleNN()
# loaded_model.load_state_dict(pytorch_model.model)
# predictions = loaded_model(torch.tensor(vector_df.limit(1).select("features").collect()[0][0].toArray(), dtype=torch.float32))

Non-obvious Insight: When using SynapseML's distributed estimators, the model_definition function runs on each Spark executor. It's crucial that this function is self-contained and doesn't rely on external variables defined outside its scope, as Spark serializes and distributes it. For PyTorch, the data_iterator provided to train_model is a partition of your Spark DataFrame, allowing each worker to process its local data chunk.

Scaling and Performance with Fabric Capacity

The performance of your deep learning workloads in Fabric Spark is directly tied to your chosen Fabric Capacity SKU and Spark configurations:

  • Fabric Capacity SKUs (F-SKUs): These determine the underlying compute resources available. For deep learning, higher F-SKUs (e.g., F64, F128, F256, F512, F1024, F2048) provide more vCores and memory, allowing for larger Spark clusters with more executors.
    • An F64 capacity, for instance, offers significantly more compute than an F2, enabling more parallel execution and larger memory footprints for your Spark driver and executors.
    • Each F-SKU translates to a specific number of Capacity Units (CUs), which in turn dictate the maximum vCores and memory available for Spark pools.
  • Spark Configurations: Fine-tuning Spark parameters is critical:
    • spark.executor.instances: Controls the number of executors. For distributed DL, this directly corresponds to the num_workers in SynapseML.
    • spark.executor.cores: Number of CPU cores per executor. More cores can speed up computation within an executor.
    • spark.executor.memory: Memory allocated to each executor. Crucial for handling large batch sizes or complex models that require significant memory.
    • spark.driver.memory: Memory for the Spark driver. Important if your driver collects large results or performs significant local processing.
  • Data Optimizations:
    • Delta Lake: Use Delta Lake format for your data in OneLake. It offers ACID transactions, schema enforcement, and performance optimizations.
    • Z-ordering & Liquid Clustering: For large Delta tables, applying Z-ordering or Liquid Clustering on frequently accessed columns (e.g., image categories, text embeddings) can dramatically reduce data scanning times, improving I/O for deep learning training.
    • Data Partitioning: Ensure your data is well-partitioned to avoid data skew and maximize parallel processing across executors.

Comparison: Raw Frameworks vs. SynapseML on Fabric Spark

When approaching deep learning on Fabric Spark, you generally have two main paths: directly using distributed features of TensorFlow/PyTorch with Spark, or leveraging SynapseML. Here's a comparison:

Feature Raw TensorFlow/PyTorch with Spark (e.g., Horovod/TorchElastic) SynapseML (DistributedTensorFlowEstimator/DistributedPyTorchEstimator)
Setup Complexity High. Requires manual orchestration of distributed processes, handling communication, and data distribution logic. Often needs custom Spark job submission scripts. Low. High-level API abstracts away distributed orchestration. Integrates seamlessly with PySpark MLlib pipeline concepts.
Distributed Training Model More direct control over distributed strategies (e.g., parameter server, all-reduce). Requires explicit code for data sharding and model synchronization. Simplified model. SynapseML manages data partitioning and uses collective communication (like all-reduce) for gradients/parameters behind the scenes.
Code Verbosity Higher. More boilerplate code for distributed setup, data loaders, and communication. Lower. Focus on defining the core model and training logic; SynapseML handles the distributed wrapper.
Integration with PySpark MLlib Limited direct integration. Often treated as a separate, external distributed process managed by Spark. Excellent. SynapseML estimators are PySpark MLlib Estimators, meaning they can be seamlessly integrated into larger ML pipelines (e.g., with FeatureTransformers, VectorAssemblers).
MLOps Integration (MLflow) Requires manual MLflow setup and logging within the distributed training loop. Easier. Can be integrated with MLflow auto-logging or custom logging within the model definition, benefiting from Fabric's native MLflow support.
Best Use Case Highly custom distributed training scenarios, bleeding-edge research, or when absolute lowest-level control over distributed communication is required. Most common deep learning tasks on Spark, especially for practitioners who want to leverage Spark's ecosystem without deep distributed systems knowledge. Ideal for productionizing DL within Fabric.

Best Practices for Deep Learning with Fabric Spark

  • Optimize Data I/O with Delta Lake: Always store your deep learning datasets in OneLake using the Delta Lake format. Leverage Delta Lake features like Z-ordering or Liquid Clustering on relevant columns (e.g., image categories, text IDs) to minimize data scan times during training.
  • Right-size Your Fabric Capacity (F-SKU): Choose an F-SKU that provides sufficient vCores and memory for your workload. For large deep learning models or datasets, start with F64 or F128 and scale up if you encounter OOM errors or slow training. Monitor Spark UI for resource utilization.
  • Tune Spark Executor Configurations: Adjust spark.executor.instances, spark.executor.cores, and spark.executor.memory based on your model size, batch size, and dataset. A common strategy is to maximize spark.executor.instances (up to the F-SKU limit) and allocate enough memory per executor to handle your batch size without OOM.
  • Leverage SynapseML for Distributed Training: For most scenarios, SynapseML's DistributedTensorFlowEstimator and DistributedPyTorchEstimator simplify distributed training significantly. Use them to abstract away the complexities of data distribution and model synchronization.
  • Implement Robust Data Preprocessing: Perform data cleaning, normalization, and feature engineering using Spark DataFrames before feeding data to your deep learning model. For text, use Hugging Face tokenizers; for images, consider resizing and augmentation.
  • Utilize MLflow for Experiment Tracking: Integrate MLflow into your deep learning workflows to track hyperparameters, metrics (loss, accuracy, F1-score), and model artifacts. This is crucial for reproducibility, comparison, and model governance.
  • Checkpoint Models Regularly: During long training runs, save model checkpoints periodically. This allows you to resume training from the last successful checkpoint in case of failures or to pick the best performing model.
  • Monitor Spark UI: Regularly check the Spark UI (accessible from your Fabric notebook or Spark job definition) to identify bottlenecks, data skew, or resource contention. Look for stages with high shuffle write/read or long garbage collection times.
  • Consider Transfer Learning: Instead of training models from scratch, leverage pre-trained models (e.g., from Hugging Face for NLP, or ImageNet pre-trained models for computer vision) and fine-tune them on your specific dataset. This significantly reduces training time and data requirements.

Common Pitfalls in Deep Learning with Fabric Spark

Pitfall Cause Fix
Out-of-Memory (OOM) Errors on Executors Insufficient spark.executor.memory for batch size, model size, or intermediate data. Increase spark.executor.memory (e.g., to 16g, 32g), reduce batch_size in your DL model, optimize data types (e.g., float16), or reduce model complexity.
Slow Training Performance Suboptimal Spark configurations, data I/O bottlenecks, data skew, or inefficient model architecture. Increase spark.executor.cores and spark.executor.instances, ensure data is Z-ordered/Liquid Clustered in Delta Lake, repartition skewed data, profile model for bottlenecks.
Data Skew Leading to Imbalanced Workload Uneven distribution of data partitions across Spark executors, causing some workers to be idle while others are overloaded. Use df.repartition(num_partitions) before training. If skew is on a specific column, consider salting or custom partitioning strategies.
Serialization Errors (e.g., PicklingError) Non-serializable objects (e.g., lambda functions capturing external scope, complex custom classes) passed to Spark UDFs or SynapseML's model_definition. Ensure all objects passed to Spark are serializable. Define functions at the top-level or use @udf with explicit type hints. Make model_definition self-contained.
Model Not Converging or Poor Performance Incorrect hyperparameters (learning rate, batch size), suboptimal model architecture, insufficient data, or data quality issues. Perform hyperparameter tuning (e.g., using Hyperopt or manual grid search), simplify/complexify model, augment data, clean and validate input data thoroughly.
Dependency Conflicts / Missing Libraries Specific versions of deep learning libraries or their dependencies are not pre-installed or conflict with existing ones. Use Fabric's workspace library management to install specific versions via wheel files or Conda environment files. Isolate environments if necessary.

  2. Tokenize Text and Prepare Features

    We'll use a pre-trained tokenizer from Hugging Face Transformers to convert text into numerical input IDs, attention masks, and token type IDs, which are required by BERT-like models. We'll wrap this in a Spark UDF.

    from transformers import AutoTokenizer
from pyspark.sql.functions import udf
from pyspark.sql.types import ArrayType, IntegerType, StructType, StructField

# Load a pre-trained tokenizer (e.g., 'bert-base-uncased')
tokenizer = AutoTokenizer.from_pretrained("bert-base-uncased")
MAX_LEN = 128 # Maximum sequence length for BERT

# Define a UDF for tokenization
@udf(returnType=StructType([
    StructField("input_ids", ArrayType(IntegerType())),
    StructField("attention_mask", ArrayType(IntegerType())),
    StructField("token_type_ids", ArrayType(IntegerType()))
]))
def tokenize_text(text: str):
    if text is None:
        return None
    encoded = tokenizer.encode_plus(
        text,
        add_special_tokens=True,
        max_length=MAX_LEN,
        padding='max_length',
        truncation=True,
        return_attention_mask=True,
        return_token_type_ids=True,
        return_tensors='pt' # Return PyTorch tensors
    )
    # Convert tensors to lists for Spark UDF compatibility
    return {
        "input_ids": encoded['input_ids'].squeeze().tolist(),
        "attention_mask": encoded['attention_mask'].squeeze().tolist(),
        "token_type_ids": encoded['token_type_ids'].squeeze().tolist()
    }

# Apply the UDF to the training DataFrame
tokenized_train_df = train_df.withColumn("encoded_features", tokenize_text(col("review_text")))
tokenized_train_df.printSchema()
tokenized_train_df.select("review_text", "encoded_features").show(2, truncate=False)

# Select relevant columns for training
final_train_df = tokenized_train_df.select(
    col("encoded_features.input_ids").alias("input_ids"),
    col("encoded_features.attention_mask").alias("attention_mask"),
    col("encoded_features.token_type_ids").alias("token_type_ids"),
    col("sentiment_label").alias("label")
)
final_train_df.show(2)

  3. Define PyTorch Model and Training Logic

    We'll define a simple PyTorch model that uses a pre-trained BERT model (from Hugging Face) as its base and adds a classification head for sentiment analysis. The training function will be passed to DistributedPyTorchEstimator.

    import torch
import torch.nn as nn
import torch.optim as optim
from torch.utils.data import Dataset, DataLoader
from transformers import AutoModelForSequenceClassification, AutoConfig
import mlflow
import numpy as np

# Define a custom PyTorch Dataset for our tokenized Spark DataFrame
class SparkTextDataset(Dataset):
    def __init__(self, data_iterator):
        self.input_ids = []
        self.attention_mask = []
        self.token_type_ids = []
        self.labels = []
        for row in data_iterator:
            self.input_ids.append(torch.tensor(row.input_ids, dtype=torch.long))
            self.attention_mask.append(torch.tensor(row.attention_mask, dtype=torch.long))
            self.token_type_ids.append(torch.tensor(row.token_type_ids, dtype=torch.long))
            self.labels.append(torch.tensor(row.label, dtype=torch.long))

    def __len__(self):
        return len(self.labels)

    def __getitem__(self, idx):
        return {
            'input_ids': self.input_ids[idx],
            'attention_mask': self.attention_mask[idx],
            'token_type_ids': self.token_type_ids[idx],
            'labels': self.labels[idx]
        }

# Define the training function that will run on each worker
def train_sentiment_model(data_iterator, context):
    # Load pre-trained BERT model for sequence classification
    # num_labels should match the number of unique sentiment labels (e.g., 2 for positive/negative, 3 for neutral)
    num_labels = 3 # Assuming 0, 1, 2 for negative, positive, neutral
    config = AutoConfig.from_pretrained("bert-base-uncased", num_labels=num_labels)
    model = AutoModelForSequenceClassification.from_pretrained("bert-base-uncased", config=config)

    optimizer = optim.AdamW(model.parameters(), lr=2e-5)
    
    # Use CrossEntropyLoss for multi-class classification
    criterion = nn.CrossEntropyLoss()

    dataset = SparkTextDataset(data_iterator)
    dataloader = DataLoader(dataset, batch_size=16, shuffle=True) # Smaller batch size for BERT

    device = torch.device("cuda" if torch.cuda.is_available() else "cpu")
    model.to(device)

    mlflow.start_run(run_name=f"PyTorch_Worker_Run_{context.worker_id}", nested=True)
    mlflow.log_param("worker_id", context.worker_id)

    for epoch in range(2): # Reduced epochs for faster demonstration
        model.train()
        total_loss = 0
        for batch in dataloader:
            inputs = {k: v.to(device) for k, v in batch.items() if k != 'labels'}
            labels = batch['labels'].to(device)

            optimizer.zero_grad()
            outputs = model(**inputs, labels=labels)
            loss = outputs.loss
            loss.backward()
            optimizer.step()
            total_loss += loss.item()
        avg_loss = total_loss / len(dataloader)
        print(f"Worker {context.worker_id} - Epoch {epoch+1} Loss: {avg_loss:.4f}")
        mlflow.log_metric(f"worker_{context.worker_id}_epoch_{epoch+1}_loss", avg_loss)
    
    mlflow.end_run()
    return model.state_dict() # Return model state dictionary

  4. Train the Model with SynapseML's DistributedPyTorchEstimator

    Now, we'll use the DistributedPyTorchEstimator to train our sentiment analysis model across the Fabric Spark cluster. We'll also integrate MLflow for tracking.

    from synapse.ml.deeplearning import DistributedPyTorchEstimator
import mlflow

# Start a parent MLflow run for the entire distributed training process
with mlflow.start_run(run_name="Distributed_Sentiment_Analysis_PyTorch"):
    mlflow.log_param("model_name", "bert-base-uncased")
    mlflow.log_param("max_sequence_length", MAX_LEN)
    mlflow.log_param("learning_rate", 2e-5)
    mlflow.log_param("epochs_per_worker", 2)
    mlflow.log_param("batch_size_per_worker", 16)
    
    # Create the DistributedPyTorchEstimator
    # num_workers: Number of Spark executors to use for training.
    # The input_cols are the columns that will be passed to the data_iterator in train_sentiment_model
    pytorch_estimator = DistributedPyTorchEstimator(
        num_workers=2, # Use 2 Spark executors for training
        model_definition=train_sentiment_model,
        input_cols=["input_ids", "attention_mask", "token_type_ids", "label"], # All columns needed by SparkTextDataset
        label_col="label", # Required by estimator, even if not directly used by model_definition
        batch_size=16, # This batch_size is for the estimator's internal data handling, not the DataLoader
        verbose=1
    )

    # Train the model
    print("Starting distributed PyTorch training...")
    pytorch_model_transformer = pytorch_estimator.fit(final_train_df)
    print("Distributed PyTorch training completed.")

    # The fit method returns a PySpark ML Transformer.
    # The actual trained PyTorch model state_dict is stored within this transformer.
    # We can save this transformer to OneLake and register it with MLflow.
    model_save_path = "Models/sentiment_bert_synapseml_model"
    pytorch_model_transformer.write().overwrite().save(model_save_path)
    mlflow.log_artifact(model_save_path, artifact_path="synapseml_model")

  5. Evaluate the Model (Post-Training)

    After training, you would apply the trained pytorch_model_transformer to your test data for predictions and evaluate its performance. This typically involves loading the saved model and performing inference.

    from pyspark.ml import PipelineModel
from pyspark.sql.functions import array_max, element_at, array_sort, struct, desc

# Load the trained SynapseML model transformer
loaded_model_transformer = PipelineModel.load("Models/sentiment_bert_synapseml_model")

# Prepare test data (tokenize it first, similar to training data)
tokenized_test_df = test_df.withColumn("encoded_features", tokenize_text(col("review_text")))

    col("encoded_features.input_ids").alias("input_ids"),
    col("encoded_features.attention_mask").alias("attention_mask"),
    col("encoded_features.token_type_ids").alias("token_type_ids"),
    col("sentiment_label").alias("label")
)

# Make predictions using the loaded model
predictions_df = loaded_model_transformer.transform(final_test_df)

# The prediction column will contain the raw output (logits) from the model.
# We need to convert these logits to predicted class labels.
# Assuming the model output is in a column named 'output' (default for SynapseML PyTorch estimator)
# and it's an array of floats representing probabilities/logits for each class.
# We need to find the index of the maximum value.

# Define a UDF to get the predicted label from the output array
@udf(returnType=IntegerType())
def get_predicted_label(output_array):
    if output_array is None:
        return None
    # Assuming output_array is a list of floats (logits or probabilities)
    return int(np.argmax(output_array))

predictions_df = predictions_df.withColumn("prediction", get_predicted_label(col("output")))

# Show some predictions
predictions_df.select("review_text", "label", "prediction", "output").show(5, truncate=False)

# Evaluate the model (e.g., accuracy)
from pyspark.ml.evaluation import MulticlassClassificationEvaluator

evaluator = MulticlassClassificationEvaluator(
    labelCol="label",
    predictionCol="prediction",
    metricName="accuracy"
)
accuracy = evaluator.evaluate(predictions_df)
print(f"Test Accuracy: {accuracy:.4f}")
