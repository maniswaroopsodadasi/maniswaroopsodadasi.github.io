Why Incremental Processing Is the Difference Between a Scalable Pipeline and a Daily Fire Drill

Imagine you're a data engineer at a mid-sized retailer. Your Fabric pipeline runs every morning, processing the previous day's transactions. On day one, it takes 4 minutes. By month six, it's taking 47 minutes — and your business stakeholders want hourly refreshes. The root cause isn't hardware. It's that your pipeline is re-reading and re-processing 180 days of data every single run to produce a result that only changed by 0.5% since the last execution.

This is the incremental processing problem, and it's one of the most impactful architectural decisions you'll make in Microsoft Fabric. Full-load pipelines are easy to reason about and simple to build, but they don't scale. Incremental pipelines require more upfront design but deliver orders-of-magnitude improvements in throughput, cost, and freshness. On Day 25 of this series, we go deep on the patterns, pitfalls, and Fabric-specific mechanics that separate amateur pipelines from production-grade data engineering.

Core Concepts: What "Incremental" Actually Means

The Spectrum of Data Loading Strategies

Incremental processing isn't a single technique — it's a family of strategies. Understanding where each fits is essential before writing a single line of code.

  • Full Load (Truncate and Reload): Deletes all existing data and reloads from source. Simple, idempotent, but computationally expensive. Appropriate only for small dimension tables under ~1 million rows or when source systems don't support change tracking.
  • Append-Only Incremental: Reads only new records since the last watermark and appends them to the target. Works when records are never updated — event logs, IoT telemetry, click streams. Fails silently when upstream data is corrected or late-arriving.
  • Upsert (Merge) Incremental: Reads changed records (new and updated) and merges them into the target using a key. Handles corrections and late arrivals. Requires a reliable change detection mechanism and a unique business key.
  • CDC (Change Data Capture): Captures row-level insert/update/delete operations at the database transaction log level. The most complete form of incremental processing. Supported natively in Fabric Data Factory pipelines via connectors to SQL Server, Azure SQL, and PostgreSQL.
  • Partition-Based Incremental: Replaces only the specific data partitions that have changed. Ideal for date-partitioned fact tables where you need to reprocess a specific day without touching the rest of the dataset.

Watermarks: The Heartbeat of Incremental Pipelines

A watermark is the bookmark that tells your pipeline "you've already processed everything up to this point." It's typically a timestamp or sequence number stored in a control table. Every incremental pipeline needs a reliable watermark strategy. The most common failure mode in incremental pipelines isn't a bug in the transformation logic — it's a corrupted or stale watermark that causes either data duplication or silent data loss.

In Microsoft Fabric, watermarks are typically stored in a Lakehouse table, a Warehouse table, or a pipeline variable persisted via a Lookup activity. Each approach has different consistency guarantees, which we'll cover in the pitfalls section.

Incremental Processing in Microsoft Fabric: The Technical Landscape

Delta Lake as the Foundation

Every Lakehouse table in Microsoft Fabric is stored as Delta Parquet format on OneLake. This is not incidental — Delta Lake's transaction log (_delta_log) is what makes sophisticated incremental patterns possible without external orchestration tools.

Delta's Change Data Feed (CDF) feature is particularly powerful. When enabled on a table, Delta tracks row-level changes (inserts, updates, deletes) in a separate change log that you can query using table_changes(). This gives you CDC-like capability entirely within your Lakehouse, without needing a source database connector.


# Enable Change Data Feed on a Delta table in Fabric Lakehouse (PySpark Notebook)
spark.sql("""
    ALTER TABLE sales.transactions
    SET TBLPROPERTIES (delta.enableChangeDataFeed = true)
""")

# Read changes since a specific version
changes_df = (
    spark.read
    .format("delta")
    .option("readChangeFeed", "true")
    .option("startingVersion", 42)   # or use startingTimestamp = "2024-11-01"
    .table("sales.transactions")
)

# Filter to only inserts and updates (exclude pre-image of updates)
incremental_df = changes_df.filter(
    changes_df["_change_type"].isin(["insert", "update_postimage"])
)

# Show the change types available
incremental_df.groupBy("_change_type").count().show()
# Output: insert, update_preimage, update_postimage, delete

A non-obvious insight: startingVersion refers to the Delta transaction log version number, not a row count or timestamp. You can retrieve the current version with DESCRIBE HISTORY tablename and store it as your watermark. This is more reliable than timestamp-based watermarks because it's immune to clock skew between systems.

Incremental Refresh in Fabric Data Factory Pipelines

Fabric Data Factory (the pipeline orchestration layer) supports incremental patterns through a combination of Lookup activities, Set Variable activities, and parameterized Copy Data activities. The canonical pattern is:

  1. Lookup activity reads the last watermark from a control table in your Lakehouse or Warehouse.
  2. Copy Data (or Dataflow Gen2) activity uses the watermark as a filter parameter in its source query.
  3. On success, a second Lookup or Script activity updates the watermark to the current run's high-water mark.

The critical architectural decision is when to update the watermark. Updating it before the data load completes means a pipeline failure leaves you with a gap. Updating it after means a failure causes a re-run that may produce duplicates if your target isn't idempotent. The correct pattern is to update the watermark only after a successful write, and to make your target write idempotent (via Delta MERGE or partition overwrite).

Incremental Refresh for Semantic Models (Power BI)

Incremental refresh in Power BI semantic models (formerly datasets) is a separate but related concept. It partitions your model's fact tables by date range and only refreshes the most recent partitions during each scheduled refresh. This is configured via Power Query's RangeStart and RangeEnd parameters and the incremental refresh policy in the model settings.

Key limits to know: Incremental refresh requires a Premium capacity (F64 or higher for full feature support, though F2 supports basic incremental refresh). The maximum number of partitions per table is 1,000. With DirectLake mode, incremental refresh behaves differently — the model reads directly from OneLake Delta files, so "refresh" means reframing which Delta snapshot is visible rather than importing data into the model's in-memory store.

Implementing a Production-Grade Upsert Pattern

The following example demonstrates a complete incremental upsert pipeline for a customer dimension table. This pattern handles new customers, updated records, and uses Delta version numbers as watermarks — a more robust approach than timestamp-based watermarks.


# Production incremental upsert pattern for customer dimension
# Fabric Lakehouse PySpark Notebook

from delta.tables import DeltaTable
from pyspark.sql import functions as F
from pyspark.sql.types import LongType

# ─── Step 1: Read current watermark from control table ───────────────────────
control_df = spark.sql("""
    SELECT last_processed_version
    FROM bronze.pipeline_control
    WHERE table_name = 'customers'
""")
last_version = control_df.collect()[0]["last_processed_version"]

# ─── Step 2: Get current Delta version of source table ───────────────────────
history_df = spark.sql("DESCRIBE HISTORY bronze.raw_customers LIMIT 1")
current_version = history_df.collect()[0]["version"]

print(f"Processing versions {last_version + 1} through {current_version}")

# ─── Step 3: Read only changed records using Delta CDF ───────────────────────
if last_version == current_version:
    print("No new versions to process. Exiting.")
    dbutils.notebook.exit("NO_NEW_DATA")

changes_df = (
    spark.read
    .format("delta")
    .option("readChangeFeed", "true")
    .option("startingVersion", last_version + 1)
    .option("endingVersion", current_version)
    .table("bronze.raw_customers")
    .filter(F.col("_change_type").isin(["insert", "update_postimage"]))
    .select(
        "customer_id",
        "full_name",
        "email",
        "country_code",
        "created_at",
        "updated_at",
        F.current_timestamp().alias("dw_updated_at")
    )
    # Deduplicate: keep the latest version of each customer in this batch
    .withColumn(
        "row_num",
        F.row_number().over(
            Window.partitionBy("customer_id")
                  .orderBy(F.col("updated_at").desc())
        )
    )
    .filter(F.col("row_num") == 1)
    .drop("row_num")
)

record_count = changes_df.count()
print(f"Records to upsert: {record_count}")

# ─── Step 4: Merge into Silver dimension table ───────────────────────────────
silver_table = DeltaTable.forName(spark, "silver.dim_customers")

(
    silver_table.alias("target")
    .merge(
        changes_df.alias("source"),
        "target.customer_id = source.customer_id"
    )
    .whenMatchedUpdate(set={
        "full_name":    "source.full_name",
        "email":        "source.email",
        "country_code": "source.country_code",
        "updated_at":   "source.updated_at",
        "dw_updated_at":"source.dw_updated_at"
    })
    .whenNotMatchedInsertAll()
    .execute()
)

# ─── Step 5: Update watermark ONLY after successful merge ────────────────────
spark.sql(f"""
    UPDATE bronze.pipeline_control
    SET last_processed_version = {current_version},
        last_run_timestamp = current_timestamp(),
        last_run_record_count = {record_count}
    WHERE table_name = 'customers'
""")

print(f"Watermark updated to version {current_version}. Pipeline complete.")

Best Practices for Incremental Processing in Fabric

Do Avoid
Use Delta transaction log version numbers as watermarks — they are monotonically increasing and immune to timezone issues Using GETDATE() or current_timestamp() as your high-water mark — clock skew between source and Fabric can cause gaps of 1-3 seconds
Store watermarks in a dedicated control table with columns for table_name, last_version, last_run_timestamp, and record_count Storing watermarks only in pipeline variables — they are lost on pipeline failure and don't survive across pipeline runs without explicit persistence
Enable Delta CDF (delta.enableChangeDataFeed = true) on all tables that will be consumed incrementally by downstream notebooks Enabling CDF retroactively on large tables without first running OPTIMIZE — CDF adds ~10-15% storage overhead and works best on well-organized files
Use replaceWhere for partition-based overwrites: .option("replaceWhere", "event_date = '2024-11-15'") Using overwrite mode without replaceWhere on partitioned tables — this replaces the entire table, not just the target partition
Add a lookback window of 2-3 days for append-only pipelines to catch late-arriving records from source systems with eventual consistency Assuming source timestamps are always accurate — ERP and CRM systems routinely backdate records by hours or days during batch imports
Run OPTIMIZE and VACUUM on Delta tables after large incremental merges to prevent small-file accumulation Running VACUUM with a retention period shorter than 7 days — this breaks time-travel queries and can corrupt CDF reads in concurrent pipelines
Use idempotent write patterns (MERGE or partition overwrite) so pipeline reruns produce the same result without manual cleanup Using append mode for upsert logic — re-running a failed pipeline will duplicate every record that was already written before the failure

Common Pitfalls

Pitfall Cause Fix
Silent data gaps after pipeline failure Watermark was updated before the write completed; next run starts from the wrong version Always update the watermark in a final step after confirming the write succeeded; use pipeline success/failure branching in Data Factory
Duplicate records in Silver layer Append mode used instead of MERGE; pipeline was rerun after partial failure Switch to Delta MERGE with a business key; add a deduplication step using row_number() partitioned by the key before merging
CDF reads fail with "version not available" error VACUUM was run with retention period shorter than the gap between pipeline runs, removing the Delta log files needed for CDF Set delta.deletedFileRetentionDuration = interval 14 days on tables using CDF; never run VACUUM with retention under 7 days
MERGE performance degrades as table grows Delta MERGE performs a full table scan when there are no partition filters on the target table Partition the target table by a date column and add a date filter to the MERGE condition: target.event_date BETWEEN date_sub(current_date(), 7) AND current_date()
Incremental refresh breaks after schema change in source New columns added to source aren't reflected in the Power Query M query used for incremental refresh Use schema evolution in Delta (mergeSchema = true) for Lakehouse tables; for semantic models, refresh the full table once after schema changes, then re-enable incremental refresh
Watermark drift on high-volume tables Processing one Delta version at a time on a table with thousands of micro-commits creates a growing backlog Process version ranges in batches (e.g., last_version to current_version) rather than one version at a time; use OPTIMIZE with auto-compaction to reduce commit frequency

Real-World Scenario: Retail Sales Pipeline with Hourly Incremental Refresh

A national retail chain needs hourly sales data in their Power BI executive dashboard. Their source is a SQL Server POS system with 50 million transactions per month across 300 stores. Here's how to build this end-to-end in Fabric:

  1. Enable CDC on the source SQL Server table. In SQL Server Management Studio, run EXEC sys.sp_cdc_enable_table @source_schema = 'dbo', @source_name = 'SalesTransactions', @role_name = NULL. This activates transaction log capture without any application changes.
  2. Create a Data Factory pipeline with a CDC source connector. In Fabric Data Factory, create a new pipeline. Add a Copy Data activity with source type "SQL Server" and enable the "Enable incremental extract" toggle. Set the CDC mode to "Transaction log" and configure the connection to your SQL Server. The connector will automatically track LSN (Log Sequence Numbers) as watermarks.
  3. Land raw CDC events into a Bronze Lakehouse table. Configure the Copy Data sink to write to a Lakehouse Delta table named bronze.pos_transactions_cdc. Set write mode to "Append" — you want to preserve the full CDC history including deletes. Enable delta.enableChangeDataFeed = true on this table for downstream consumption.
  4. Create a PySpark notebook for Silver layer transformation. This notebook reads from the Bronze CDF, applies business logic (tax calculation, store dimension lookup, currency normalization), deduplicates using transaction_id as the key, and MERGEs into silver.fact_sales. The notebook reads the watermark from bronze.pipeline_control and updates it on completion.
  5. Partition the Silver fact table by transaction_date. When creating silver.fact_sales, partition by transaction_date using spark.sql("CREATE TABLE silver.fact_sales ... PARTITIONED BY (transaction_date DATE)"). This reduces MERGE scan cost from full-table to single-day partition — a 365x reduction in data scanned for daily updates.
  6. Schedule the pipeline to run every hour. In Fabric Data Factory, add a Schedule trigger set to every 60 minutes starting at :05 past the hour (giving the source system time to commit its hourly batch). Set a pipeline timeout of 45 minutes to prevent overlapping runs.
  7. Configure the semantic model with incremental refresh. In Power BI Desktop, add RangeStart and RangeEnd parameters (DateTime type) to the M query. Filter the transaction_date column using these parameters. In the incremental refresh policy, set "Store rows in the last 2 years" and "Refresh rows in the last 3 days." Publish to Fabric workspace on F64 or higher capacity.
  8. Validate end-to-end latency. Monitor the pipeline run history in Fabric's monitoring hub. Target: CDC extraction ≤ 3 minutes, Silver MERGE ≤ 8 minutes, semantic model refresh ≤ 5 minutes. Total latency from POS transaction to Power BI dashboard: under 20 minutes, compared to the original 47-minute full-load batch.

Key Takeaways

  • Delta transaction log versions are superior watermarks to timestamps — they are monotonically increasing, immune to clock skew, and directly supported by Delta CDF's startingVersion parameter.
  • Always update watermarks after successful writes, never before. Use Data Factory's success/failure branching to ensure the control table is only updated when the data write completes without error.
  • Delta Change Data Feed gives you CDC capability within your Lakehouse without needing a source database connector — enabling incremental propagation between Bronze, Silver, and Gold layers entirely within Fabric.
  • Partitioning is not optional for large fact tables — a Delta MERGE on an unpartitioned 500M-row table will scan every file. Adding a date partition filter to the MERGE condition reduces this to the files in the target partition only.
  • Idempotency is a requirement, not a nice-to-have — every incremental pipeline will eventually fail and be rerun. If your pipeline isn't idempotent, a rerun produces incorrect data silently, which is worse than a visible failure.
  • DirectLake semantic models behave differently from import mode during incremental refresh — they read Delta snapshots directly from OneLake, so "refresh" is a metadata operation, not a data copy. This makes them dramatically faster but requires the underlying Delta table to be well-maintained with regular OPTIMIZE runs.
  • The lookback window is your insurance policy against late-arriving data — always process a 2-3 day lookback window for append-only pipelines consuming from operational systems with eventual consistency guarantees.
