Why Cross-Platform Analytics Integration Is Now a Business Imperative

Most enterprise data estates are not monolithic. A typical Fortune 500 company runs Snowflake for its data science team, Azure Synapse for legacy ETL pipelines, Databricks for ML feature engineering, Google BigQuery for a recently acquired subsidiary, and Power BI Premium for executive dashboards — all simultaneously. When Microsoft Fabric entered general availability in November 2023, it introduced a compelling unification story, but the reality is that organizations cannot — and often should not — migrate everything overnight.

The real business problem is this: your CFO needs a consolidated P&L report that pulls actuals from Snowflake, forecasts from a Databricks ML model, and budget data from an on-premises SQL Server, all refreshed by 6 AM every trading day. Building that pipeline naively means brittle point-to-point connectors, duplicated data, stale caches, and a 3 AM on-call rotation. Cross-platform analytics integration in Microsoft Fabric is the architectural discipline that eliminates that fragility — and Day 60 is where we go deep on how to actually build it.

Core Concepts: The Fabric Integration Architecture

OneLake as the Universal Landing Zone

OneLake is the foundational insight that makes cross-platform integration tractable. Every Microsoft Fabric tenant gets exactly one OneLake — a single logical data lake built on Azure Data Lake Storage Gen2 — and every Fabric workload (Lakehouse, Warehouse, KQL Database, Eventstream) reads and writes Delta Parquet files stored there. This means that data ingested from an external platform lands once in OneLake and is immediately queryable by every Fabric engine without copying.

The non-obvious insight: OneLake supports ADLS Gen2 shortcut mounting, which means your existing Azure Data Lake Storage accounts appear as first-class folders inside OneLake without any data movement. The same mechanism works for Amazon S3 and Google Cloud Storage buckets via the OneLake Shortcuts feature. When you create an S3 shortcut, Fabric reads the Parquet files directly from S3 at query time — there is no ETL, no copy, no sync job. The latency cost is real (cross-cloud egress adds 20–80ms per file open), but for analytical workloads querying large files, this is negligible compared to the operational simplicity gained.

DirectLake Mode and Its Cross-Platform Implications

DirectLake is the Power BI storage mode that reads Delta Parquet files from OneLake directly into the Analysis Services engine's in-memory columnar store, bypassing the traditional Import/DirectQuery dichotomy. For cross-platform integration, this matters because any data you land in OneLake — regardless of its origin (Snowflake export, Databricks write, Kafka stream via Eventstream) — becomes immediately available to Power BI semantic models in DirectLake mode without a scheduled dataset refresh.

The critical limit to know: DirectLake falls back to DirectQuery automatically when a table exceeds the guardrails for your Fabric SKU. On F2, the limit is 300 million rows per table. On F64, it is 1.5 billion rows. On F256 and above, there is no documented row limit. If your cross-platform pipeline lands a table that exceeds these thresholds, your report silently degrades to DirectQuery performance — a pitfall that catches many teams off guard in production.

Fabric Data Pipelines vs. Fabric Dataflows Gen2

Fabric offers two orchestration primitives for cross-platform ingestion. Data Pipelines (based on Azure Data Factory v2 runtime) are the right choice for high-volume, scheduled, or event-triggered movement of data from external systems — they support 90+ connectors including Snowflake, Databricks Delta Sharing, Google BigQuery, SAP HANA, and Salesforce. Dataflows Gen2 (Power Query Online runtime) are better for analyst-owned, low-code transformations where the source data is under 1 GB per query and the team lacks Spark expertise.

The architectural rule: use Data Pipelines for ingestion orchestration and Dataflows Gen2 for the last-mile transformation that business analysts own. Never use Dataflows Gen2 as your primary ingestion mechanism for external systems at scale — the Power Query Online engine has a 30-minute timeout per query and does not support incremental watermark patterns natively.

Technical Deep Dive: Integration Patterns

Pattern 1 — Snowflake to Fabric via Delta Sharing

Snowflake supports the open Delta Sharing protocol, which allows a Snowflake account to publish Delta tables as a read-only share that any Delta Sharing-compatible reader can consume. Microsoft Fabric's Lakehouse supports Delta Sharing as a shortcut source. The practical steps are: enable Delta Sharing on the Snowflake side, generate a share profile JSON, and create a Fabric shortcut pointing to that share. The data never leaves Snowflake's storage — Fabric reads the Delta log and Parquet files directly via HTTPS.

The catch: Delta Sharing from Snowflake does not support Change Data Feed (CDF). You get full snapshot reads only. For incremental patterns, you must use Snowflake's COPY INTO to export to an S3 bucket and then mount that bucket as an OneLake shortcut, or use a Fabric Data Pipeline with the Snowflake connector and a watermark column.

Pattern 2 — Databricks to Fabric via Unity Catalog Mirroring

As of mid-2024, Microsoft Fabric introduced Mirroring as a near-real-time replication feature. Fabric Mirroring for Azure Databricks reads the Delta transaction log of a Unity Catalog-managed table and replicates changes into OneLake continuously, with typical latency under 60 seconds. This is the highest-fidelity integration pattern for Databricks workloads because it preserves the Delta format, schema evolution, and partition structure without any transformation.

Mirroring requires the Databricks workspace to be in the same Azure region as the Fabric capacity for optimal performance. Cross-region mirroring works but incurs Azure inter-region data transfer costs and adds 2–5 seconds of additional latency per transaction batch.

Pattern 3 — Google BigQuery via Fabric Data Pipeline

BigQuery does not support Delta Sharing or any Fabric-native protocol. The integration path is a Fabric Data Pipeline using the BigQuery connector (OAuth 2.0 service account authentication), which reads BigQuery tables via the BigQuery Storage Read API and writes Delta Parquet to OneLake. The BigQuery Storage Read API is significantly faster than the legacy REST API — it uses Apache Arrow columnar format over gRPC and can sustain 1–2 GB/s throughput on large tables.

Below is a PySpark notebook activity you can embed in a Fabric Data Pipeline to perform an incremental load from BigQuery, using a watermark stored in a Fabric Lakehouse Delta table:


# Fabric Spark Notebook — BigQuery Incremental Load
# Requires: spark-bigquery-with-dependencies JAR (pre-installed in Fabric Spark runtime 3.4+)

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, max as spark_max
from delta.tables import DeltaTable
import datetime

spark = SparkSession.builder.getOrCreate()

LAKEHOUSE_PATH = "abfss://analytics@onelake.dfs.fabric.microsoft.com/FinanceLakehouse.Lakehouse"
WATERMARK_TABLE = f"{LAKEHOUSE_PATH}/Tables/bq_watermarks"
TARGET_TABLE    = f"{LAKEHOUSE_PATH}/Tables/bq_sales_transactions"
BQ_PROJECT      = "my-gcp-project"
BQ_DATASET      = "finance_prod"
BQ_TABLE        = "sales_transactions"

# Read current watermark
try:
    wm_df = spark.read.format("delta").load(WATERMARK_TABLE)
    last_loaded = wm_df.filter(col("table_name") == BQ_TABLE) \
                       .select("last_loaded_ts").collect()[0][0]
except Exception:
    last_loaded = datetime.datetime(2020, 1, 1)

print(f"Loading records updated after: {last_loaded}")

# Read from BigQuery using Storage Read API
bq_df = (
    spark.read.format("bigquery")
    .option("project", BQ_PROJECT)
    .option("dataset", BQ_DATASET)
    .option("table", BQ_TABLE)
    .option("filter", f"updated_at > '{last_loaded}'")
    .option("readDataFormat", "ARROW")
    .load()
)

record_count = bq_df.count()
print(f"Records fetched: {record_count}")

if record_count > 0:
    # Upsert into Delta table using merge
    if DeltaTable.isDeltaTable(spark, TARGET_TABLE):
        delta_target = DeltaTable.forPath(spark, TARGET_TABLE)
        delta_target.alias("target").merge(
            bq_df.alias("source"),
            "target.transaction_id = source.transaction_id"
        ).whenMatchedUpdateAll() \
         .whenNotMatchedInsertAll() \
         .execute()
    else:
        bq_df.write.format("delta").mode("overwrite").save(TARGET_TABLE)

    # Update watermark
    new_wm = bq_df.select(spark_max("updated_at").alias("last_loaded_ts")) \
                  .withColumn("table_name", col("last_loaded_ts").cast("string"))
    # Simplified watermark upsert
    new_wm_data = [(BQ_TABLE, bq_df.select(spark_max("updated_at")).collect()[0][0])]
    wm_schema = ["table_name", "last_loaded_ts"]
    spark.createDataFrame(new_wm_data, wm_schema) \
         .write.format("delta").mode("overwrite") \
         .option("replaceWhere", f"table_name = '{BQ_TABLE}'") \
         .save(WATERMARK_TABLE)

print("BigQuery incremental load complete.")

Pattern 4 — Real-Time Streams via Eventstream

For platforms emitting events (Confluent Kafka, Azure Event Hubs, AWS Kinesis via custom source), Fabric Eventstream provides a no-code streaming pipeline that lands events into a KQL Database (for sub-second analytics) or a Lakehouse Delta table (for batch analytics). The Eventstream processor supports windowed aggregations, schema enforcement, and fan-out to multiple destinations simultaneously.

Cross-Workspace Query Federation with T-SQL

One of the most powerful and underused features in Fabric is the ability to query across Lakehouses and Warehouses within the same tenant using the Fabric SQL endpoint. Every Lakehouse automatically exposes a read-only SQL Analytics Endpoint, and you can reference tables across workspaces using three-part naming in T-SQL. Below is a practical example that joins a Snowflake-sourced sales table in one Lakehouse with a Databricks-sourced customer table in another:


-- Fabric Warehouse T-SQL — Cross-Lakehouse federated query
-- Both Lakehouses are in the same Fabric tenant
-- Use the SQL Analytics Endpoint connection string for each Lakehouse

-- Step 1: Create external references using three-part naming
-- Format: [WorkspaceName].[LakehouseName].[SchemaName].[TableName]
-- Note: Cross-workspace references require the caller to have Viewer role on both workspaces

SELECT
    s.transaction_id,
    s.sale_date,
    s.gross_revenue_usd,
    s.product_sku,
    c.customer_segment,
    c.country_iso2,
    c.lifetime_value_band,
    p.product_category,
    p.margin_pct
FROM
    -- Snowflake-sourced sales data (landed via Data Pipeline)
    [FinanceWorkspace].[SnowflakeLakehouse].[dbo].[sales_transactions] AS s

INNER JOIN
    -- Databricks-sourced customer master (landed via Mirroring)
    [CustomerWorkspace].[DatabricksLakehouse].[dbo].[customer_master] AS c
    ON s.customer_id = c.customer_id

INNER JOIN
    -- Product catalog managed natively in Fabric Warehouse
    [ProductWorkspace].[ProductWarehouse].[dbo].[product_catalog] AS p
    ON s.product_sku = p.sku

WHERE
    s.sale_date >= DATEADD(day, -90, GETDATE())
    AND c.is_active = 1
    AND s.gross_revenue_usd > 0

ORDER BY
    s.sale_date DESC,
    s.gross_revenue_usd DESC;
Architecture Note: Cross-workspace T-SQL queries in Fabric execute via the Fabric SQL engine's distributed query processor. Each referenced Lakehouse's SQL Analytics Endpoint processes its predicate pushdown locally, and only the filtered result sets are shuffled for the join. For joins between large tables (over 100M rows each), add explicit partition filter predicates on both sides to avoid full table scans across the network boundary.

Best Practices for Cross-Platform Integration

Do Avoid
Use OneLake Shortcuts for read-heavy, infrequently updated external data — eliminates ETL entirely Using Shortcuts for high-frequency write workloads — S3/ADLS shortcut reads are eventually consistent and not suitable for streaming targets
Store watermark state in a Delta table within the same Lakehouse as the target — survives pipeline restarts and is queryable for monitoring Storing watermarks in pipeline parameters or Azure Key Vault secrets — these are not auditable and break incremental logic on pipeline clone/copy
Use Fabric Mirroring for Databricks Unity Catalog tables where latency under 5 minutes is required Using scheduled Data Pipelines with 5-minute intervals as a substitute for Mirroring — this creates excessive Spark job overhead and Snowflake/Databricks compute costs
Partition external Delta tables by date (e.g., year/month/day) before landing in OneLake — enables partition pruning in both Spark and SQL Analytics Endpoint Landing unpartitioned large tables into OneLake — a 500GB unpartitioned Parquet file forces full file scans on every query, negating DirectLake performance
Assign a dedicated Fabric capacity (F4 minimum) to cross-platform integration workloads — prevents noisy-neighbor throttling from BI workloads on shared F2 capacity Running heavy ingestion pipelines on the same F2 capacity as interactive Power BI reports — pipeline Spark jobs consume CUs aggressively and trigger throttling for report users
Use Fabric's built-in lineage view (Workspace Settings → Lineage) to document cross-platform data flows for governance and impact analysis Relying solely on pipeline documentation in README files — these go stale within weeks and are invisible to data governance tools like Microsoft Purview

Common Pitfalls in Cross-Platform Integration

Pitfall Cause Fix
DirectLake semantic model silently falls back to DirectQuery Source Delta table exceeds SKU row limit (e.g., 300M rows on F2) or contains unsupported Delta features like Deletion Vectors Run ANALYZE TABLE in Spark to update Delta statistics; upgrade to F64+ for larger tables; disable Deletion Vectors on tables used by DirectLake models
OneLake S3 Shortcut returns stale data S3 source uses eventual consistency for list operations; Fabric caches the file list for up to 5 minutes Add a _delta_log transaction log to S3 data (convert to Delta format on the source side) so Fabric uses the Delta log for file discovery instead of S3 list API
Fabric Mirroring stops replicating after Databricks schema change Mirroring pauses on breaking schema changes (column drops, type changes) and requires manual resume Enable schema evolution alerts in Databricks Unity Catalog; build a monitoring pipeline that checks Mirroring status via the Fabric REST API /v1/workspaces/{workspaceId}/mirroredDatabases/{mirroredDatabaseId} every 15 minutes
Cross-workspace T-SQL query times out after 10 minutes Default SQL Analytics Endpoint query timeout is 10 minutes; large cross-workspace joins exceed this Pre-aggregate data in the source Lakehouse using a scheduled Spark notebook; materialize join results as a Delta table in the consuming workspace rather than querying live
Dataflow Gen2 fails on BigQuery source with 30-minute timeout Power Query Online engine timeout; large BigQuery tables exceed the limit Replace Dataflow Gen2 with a Fabric Data Pipeline using the BigQuery connector and a Spark notebook activity for transformation

Real-World Scenario: Building a Unified CFO Dashboard Across Four Platforms

A global retail company needs a daily consolidated P&L dashboard in Power BI that combines: actuals from Snowflake (finance team's system of record), ML-based forecasts from Databricks, marketing spend from Google BigQuery, and inventory data from an on-premises SQL Server 2019 instance. The target is a 6 AM refresh SLA with data no older than 2 hours.

  1. Provision a dedicated F8 Fabric capacity in the same Azure region (East US 2) as the Snowflake account and Databricks workspace. Assign the integration workspace to this capacity to isolate ingestion CU consumption from the BI workspace on F4.
  2. Configure Fabric Mirroring for Databricks: In the Fabric portal, create a new Mirrored Database pointing to the Unity Catalog schema ml_prod.forecast_outputs. Mirroring begins replicating the daily_sku_forecast Delta table into OneLake within 5 minutes of setup. No pipeline required — the table is live in the Fabric Lakehouse immediately.
  3. Create an OneLake S3 Shortcut for the BigQuery export bucket: The BigQuery team runs a daily scheduled query that exports the marketing_spend_daily table to an S3 bucket in Parquet format at 3 AM UTC. Create a Fabric OneLake shortcut pointing to s3://corp-analytics-exports/marketing_spend/. The data is queryable in Fabric without any copy job.
  4. Build a Fabric Data Pipeline for Snowflake actuals: Create a pipeline with a Snowflake connector activity reading from FINANCE_DB.PUBLIC.GL_TRANSACTIONS with a watermark on POSTED_DATE >= @{pipeline().parameters.last_run_date}. Write output to the integration Lakehouse Delta table gl_transactions_delta. Schedule at 4 AM UTC.
  5. Configure the On-Premises Gateway for SQL Server: Install the Fabric On-Premises Data Gateway on a VM in the corporate data center. Create a Data Pipeline with a SQL Server connector using the gateway, reading the inventory.dbo.stock_levels table with a last_updated watermark. Schedule at 4:30 AM UTC.
  6. Build the consolidation Spark notebook: A Fabric Spark notebook (scheduled at 5 AM UTC via a pipeline) reads all four sources from the integration Lakehouse, applies currency conversion (EUR/GBP to USD using a rates table stored in Fabric Warehouse), and writes the consolidated pnl_consolidated Delta table partitioned by fiscal_year/fiscal_month/region.
  7. Create the DirectLake Power BI semantic model: Point a new semantic model at the pnl_consolidated table in DirectLake mode. Verify the table row count (approximately 45M rows) is within the F8 DirectLake limit of 600M rows. Build the CFO dashboard with DAX measures for gross margin, EBITDA, and forecast variance.
  8. Implement monitoring: Create a Fabric Data Pipeline that runs at 5:55 AM UTC, queries the Delta table's _delta_log to verify the latest commit timestamp is within the last 2 hours, and sends an alert via Logic Apps to the data engineering Slack channel if the SLA is at risk.
  9. Validate the 6 AM SLA: End-to-end pipeline run time is approximately 45 minutes (Snowflake: 12 min, SQL Server: 8 min, consolidation Spark: 18 min, semantic model cache warm: 7 min). The F8 capacity provides sufficient CUs to run all pipelines concurrently without throttling.

Key Takeaways

  • OneLake Shortcuts are your first tool of choice for external data that is already in Parquet or Delta format — they eliminate ETL entirely and make data immediately queryable by all Fabric engines without copying a single byte.
  • Fabric Mirroring for Databricks Unity Catalog provides sub-60-second replication latency with zero pipeline code — it is the correct integration pattern for any Databricks table that changes more frequently than every 15 minutes.
  • DirectLake mode has hard row limits per SKU — always verify your table size against the F2/F64/F256 thresholds before building a semantic model, and disable Deletion Vectors on Delta tables used as DirectLake sources.
  • Cross-workspace T-SQL federation works in Fabric but requires careful predicate pushdown design — materialize large join results as Delta tables rather than relying on live cross-workspace queries for production dashboards.
  • Watermark state belongs in Delta tables, not pipeline parameters — this makes your incremental load logic auditable, restartable, and visible to monitoring queries without any additional infrastructure.
  • Separate your integration capacity from your BI capacity — Spark ingestion jobs are CU-intensive and will throttle interactive Power BI users if they share an F2 or F4 capacity.
  • The BigQuery Storage Read API with Arrow format is 10–20x faster than the legacy REST API for large table exports — always specify readDataFormat=ARROW in your Spark BigQuery connector configuration.
