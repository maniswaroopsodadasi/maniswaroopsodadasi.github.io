When the Lights Go Out: Why Fabric DR Planning Is Not Optional

In March 2024, a major European retailer running their entire analytics platform on a cloud data lakehouse suffered a regional outage during peak trading hours. Their BI dashboards went dark, their data pipelines stalled mid-execution, and their data science teams lost access to feature stores right before a critical model retraining cycle. The recovery took eleven hours — not because the cloud provider failed to restore infrastructure, but because the organization had no documented recovery runbook for their data platform.

Microsoft Fabric changes the economics and architecture of analytics dramatically, but it does not eliminate the need for disaster recovery planning — it simply changes what that planning looks like. Fabric is a SaaS platform built on Azure infrastructure, which means Microsoft handles hardware redundancy, but you are still responsible for data durability, workspace recoverability, pipeline continuity, and RTO/RPO targets at the application layer. This distinction is the single most misunderstood aspect of Fabric DR planning among teams migrating from on-premises data warehouses.

Today, on Day 40 of this series, we go deep into Fabric Disaster Recovery: what the platform provides natively, what you must build yourself, and how to architect a resilient analytics platform that can survive regional failures, accidental deletions, and configuration corruption.

Understanding the Fabric Shared Responsibility Model

What Microsoft Manages

Microsoft Fabric runs on Azure infrastructure with built-in redundancy at the compute and storage layers. OneLake, the unified storage layer underpinning all Fabric workloads, stores data using Locally Redundant Storage (LRS) by default, with Geo-Redundant Storage (GRS) available depending on your capacity region and configuration. Microsoft guarantees infrastructure availability through their standard Azure SLA, but this does not cover your workspace configurations, pipeline definitions, semantic model metadata, or the logical integrity of your Delta Parquet tables.

What You Must Manage

  • Workspace and item metadata: Lakehouse schemas, Warehouse DDL, Dataflow Gen2 definitions, and Pipeline JSON are not automatically backed up to a secondary region — you must export and version-control these artifacts.
  • Delta table transaction logs: The _delta_log directory in OneLake is critical for time-travel and recovery. Accidental deletion of this folder destroys your table history even if the Parquet files remain.
  • Semantic model definitions: Power BI semantic models (formerly datasets) in DirectLake mode store their metadata in the workspace, not in OneLake. A workspace corruption event can orphan your reports even if the underlying Delta tables are intact.
  • Capacity-level configuration: Spark compute pools, custom environments, and notebook configurations are workspace-scoped and require separate backup strategies.
  • Cross-workspace dependencies: Shortcuts pointing to external ADLS Gen2 or S3 sources, and cross-workspace semantic model references, can silently break during a DR event if not mapped and tested.

Core DR Concepts Applied to Microsoft Fabric

RTO and RPO in the Fabric Context

Recovery Time Objective (RTO) is how long your platform can be unavailable before business impact becomes unacceptable. Recovery Point Objective (RPO) is how much data loss is tolerable — measured in time. For a Fabric-based analytics platform, these targets must be defined per workload tier, not as a single platform-wide number.

Workload Tier Typical RTO Target Typical RPO Target Fabric Components Affected
Executive BI Dashboards 30 minutes 4 hours Semantic models, Power BI reports, DirectLake connections
Operational Data Pipelines 2 hours 1 hour Data Factory pipelines, Dataflow Gen2, Lakehouse tables
Data Science / ML Workflows 4 hours 24 hours Notebooks, ML experiments, Spark environments
Historical Data Archive 24 hours 7 days OneLake Delta tables, Warehouse snapshots

Fabric Capacity and Regional Considerations

Fabric capacities (F2 through F2048) are provisioned in a specific Azure region. If that region experiences a full outage, your capacity becomes unavailable. Microsoft does not automatically fail over your Fabric capacity to another region — this is a critical architectural decision you must make upfront. The two primary strategies are:

  • Active-Passive: A primary capacity in Region A with a standby capacity in Region B. OneLake data is replicated via Azure Data Factory or Fabric pipelines. The secondary capacity is kept warm but not actively serving traffic. Lower cost, higher RTO (typically 2-4 hours for full failover).
  • Active-Active: Two capacities in different regions both serving traffic, with data synchronized bidirectionally. Higher cost (you pay for two capacities), but RTO can be under 15 minutes with proper DNS and gateway routing.

OneLake Data Protection Strategies

Delta Table Time Travel as a First Line of Defense

Delta Lake's time-travel capability, built into every Fabric Lakehouse table, is your most immediate recovery tool for accidental data corruption or bad pipeline runs. By default, Fabric retains Delta transaction log history for 7 days, but this is configurable per table using the delta.logRetentionDuration and delta.deletedFileRetentionDuration table properties.


# PySpark: Configure extended retention and restore a table to a prior state
# Run this in a Fabric Notebook against your Lakehouse

from delta.tables import DeltaTable
from pyspark.sql import SparkSession

spark = SparkSession.builder.getOrCreate()

# Set extended retention on a critical table (30 days)
spark.sql("""
    ALTER TABLE lakehouse_gold.sales_fact
    SET TBLPROPERTIES (
        'delta.logRetentionDuration' = 'interval 30 days',
        'delta.deletedFileRetentionDuration' = 'interval 30 days'
    )
""")

# Inspect available history versions
delta_table = DeltaTable.forName(spark, "lakehouse_gold.sales_fact")
history_df = delta_table.history()
history_df.select("version", "timestamp", "operation", "operationParameters") \
          .orderBy("version", ascending=False) \
          .show(10, truncate=False)

# Restore to a known-good version (e.g., version 42 before a bad pipeline run)
delta_table.restoreToVersion(42)

# Alternatively, restore to a specific timestamp
# delta_table.restoreToTimestamp("2024-11-15 08:00:00")

print("Table restored successfully. Validating row count...")
spark.sql("SELECT COUNT(*) as row_count FROM lakehouse_gold.sales_fact").show()

Non-obvious insight: When you call RESTORE on a Delta table in Fabric, it creates a new transaction log entry rather than rolling back the log. This means your audit trail is preserved, but it also means the "restored" state is version N+1, not a true rollback. If you need to prove data lineage to auditors, this distinction matters significantly.

Cross-Region Replication Using Fabric Pipelines

For true regional DR, you need to replicate OneLake data to a secondary region. The most Fabric-native approach uses a scheduled Copy Activity in a Data Factory pipeline within Fabric, writing to an ADLS Gen2 account in your DR region. This secondary storage can then be mounted as a Shortcut in your standby Fabric workspace.


# Python script to automate DR replication validation
# Run as a Fabric Notebook scheduled daily to verify replication lag

import requests
import json
from datetime import datetime, timezone
from notebookutils import mssparkutils

# Configuration
PRIMARY_LAKEHOUSE_PATH = "abfss://primary-workspace@onelake.dfs.fabric.microsoft.com/gold_lakehouse.Lakehouse/Tables/sales_fact"
DR_STORAGE_PATH = "abfss://dr-container@drstorage.dfs.core.windows.net/fabric-dr/gold_lakehouse/Tables/sales_fact"

def get_latest_delta_version(path: str) -> dict:
    """Read the latest Delta log entry to determine version and timestamp."""
    try:
        log_path = f"{path}/_delta_log"
        # List log files and find the latest checkpoint or JSON log
        files = mssparkutils.fs.ls(log_path)
        json_logs = sorted(
            [f for f in files if f.name.endswith('.json')],
            key=lambda x: x.name,
            reverse=True
        )
        if not json_logs:
            return {"version": -1, "timestamp": None}
        
        latest_log = json_logs[0]
        content = mssparkutils.fs.head(latest_log.path, 4096)
        log_entry = json.loads(content.split('\n')[0])
        
        return {
            "version": int(latest_log.name.replace('.json', '')),
            "timestamp": log_entry.get('commitInfo', {}).get('timestamp', None)
        }
    except Exception as e:
        return {"version": -1, "timestamp": None, "error": str(e)}

primary_state = get_latest_delta_version(PRIMARY_LAKEHOUSE_PATH)
dr_state = get_latest_delta_version(DR_STORAGE_PATH)

version_lag = primary_state["version"] - dr_state["version"]
print(f"Primary version: {primary_state['version']}")
print(f"DR version: {dr_state['version']}")
print(f"Version lag: {version_lag} commits")

if version_lag > 3:
    print("WARNING: DR replication lag exceeds threshold. Triggering alert.")

else:
    print("DR replication status: HEALTHY")

Workspace and Artifact Backup Strategies

Git Integration as a Configuration Backup Mechanism

Fabric's Git integration (available for workspaces connected to Azure DevOps or GitHub) is your primary mechanism for backing up workspace artifact definitions. When enabled, Fabric serializes Notebooks, Pipelines, Dataflow Gen2 definitions, Lakehouse metadata, and Warehouse DDL scripts into a Git repository. This gives you version-controlled, point-in-time recovery for your platform configuration — not your data, but the code and metadata that defines how data flows.

Critical limitation: Semantic model (Power BI dataset) PBIX files and report PBIX files are not fully serialized by Fabric Git integration as of early 2025. You must separately export these using the Power BI REST API and store them in your repository. Automate this with a scheduled Fabric notebook calling the Power BI Admin API.

Fabric REST API for Workspace Inventory

Before you can back up a workspace, you need a complete inventory of its contents. The Fabric REST API (https://api.fabric.microsoft.com/v1) exposes endpoints for listing all items, their types, and their definitions. Integrate this into your DR runbook to generate a pre-failover inventory report.

Best Practices for Fabric DR Planning

Do Avoid
Connect every production workspace to Git (Azure DevOps or GitHub) and enforce commit policies requiring PR reviews for pipeline changes Relying solely on OneLake's built-in redundancy as your DR strategy — LRS protects against hardware failure, not regional outages or accidental deletion
Set delta.logRetentionDuration to at least 30 days on Gold-layer tables that feed executive reporting Running VACUUM with a retention period shorter than your RPO — this permanently destroys the Parquet files needed for time-travel recovery
Test your DR runbook quarterly with a full failover drill to a secondary workspace in a different region Assuming that restoring a workspace from Git is sufficient — Git restores code, not data; you need both a code restore and a data restore procedure
Use Fabric Mirroring for operational databases (Azure SQL, Cosmos DB) to maintain a near-real-time replica in OneLake with sub-minute latency Building custom CDC pipelines from scratch when Fabric Mirroring provides managed, low-latency replication with automatic schema evolution
Document all Shortcut dependencies in a central catalog — shortcuts to external ADLS or S3 sources will break during failover if the target storage is also in the failed region Creating Shortcuts to storage accounts in the same region as your primary Fabric capacity — this creates a single point of failure for both compute and storage
Implement workspace-level monitoring using Fabric Capacity Metrics app to detect anomalous pipeline failures that may indicate data corruption before it propagates to Gold layer Treating DR as a one-time setup task — Fabric releases new features monthly, and your DR procedures must be reviewed after each major platform update

Common Pitfalls in Fabric DR Planning

Pitfall Cause Fix
VACUUM destroys time-travel recovery window Default VACUUM retention is 7 days, and teams run it aggressively to control storage costs without checking RPO requirements Set a table-level VACUUM policy: VACUUM sales_fact RETAIN 720 HOURS and enforce via a Fabric Data Quality rule that alerts if retention drops below RPO threshold
Semantic models orphaned after workspace restore DirectLake semantic models store connection metadata (workspace ID, Lakehouse item ID) that changes when a workspace is recreated from Git After workspace restore, use the Power BI REST API PATCH /datasets/{datasetId}/Default.UpdateDatasources to rebind semantic models to the new Lakehouse item IDs
Pipeline credentials not captured in Git backup Fabric Git integration serializes pipeline structure but not connection credentials or linked service secrets stored in Azure Key Vault references Maintain a separate Key Vault DR procedure; document all connection names and their Key Vault secret mappings in a runbook stored outside Fabric
Dataflow Gen2 staging Lakehouse not replicated Dataflow Gen2 uses an internal staging Lakehouse that is workspace-scoped and not visible in the workspace item list — teams forget to include it in DR scope Identify the staging Lakehouse via the Fabric REST API (GET /workspaces/{workspaceId}/items?type=Lakehouse) and include it explicitly in your replication pipeline
Capacity unavailability mistaken for data loss When a Fabric capacity goes offline, workspaces appear inaccessible, leading teams to trigger DR procedures unnecessarily and causing data consistency issues Implement a two-stage DR trigger: first confirm capacity status via Azure Service Health API before initiating failover; set a minimum 30-minute wait period for transient outages
Cross-workspace semantic model references break during failover Reports in Workspace B referencing a shared semantic model in Workspace A will fail if Workspace A is in the affected region and Workspace B is in the DR region Maintain a dependency map of all cross-workspace semantic model references; during DR, restore Workspace A first and validate semantic model availability before restoring dependent workspaces

Real-World Scenario: Executing a Fabric DR Failover

The following scenario walks through a complete DR failover for a retail analytics platform running on Fabric F64 capacity in East US, failing over to a warm standby F32 capacity in West US 2.

  1. Detect the incident (T+0 minutes): Azure Service Health dashboard shows East US experiencing a partial outage affecting Fabric capacity fabric-prod-eastus-f64. The on-call data engineer receives a PagerDuty alert triggered by a failed Fabric pipeline heartbeat notebook that runs every 15 minutes.
  2. Confirm scope and avoid premature failover (T+5 minutes): Query the Azure Resource Health API to confirm the capacity is degraded, not just experiencing a transient spike. Check the Fabric Capacity Metrics app (if accessible from a secondary region browser session) to see if CU utilization shows a flatline — confirming the capacity is offline, not overloaded.
  3. Activate the DR workspace (T+15 minutes): The DR workspace retail-analytics-dr-westus2 is pre-configured in West US 2 on capacity fabric-dr-westus2-f32. All Lakehouses in this workspace have Shortcuts pointing to the DR ADLS Gen2 account (drstorage.dfs.core.windows.net) which receives hourly replication from the primary OneLake via a scheduled Fabric pipeline. Verify the latest replication timestamp by running the validation notebook from the code example above.
  4. Assess data lag (T+20 minutes): The validation notebook reports a version lag of 2 commits on the sales_fact table, representing approximately 45 minutes of data. This is within the 1-hour RPO defined for operational pipelines. Document the exact lag in the incident log for post-mortem analysis.
  5. Restore workspace configuration from Git (T+25 minutes): The DR workspace Git integration is already connected to the same Azure DevOps repository as the primary workspace. Sync the workspace to the latest commit on the main branch. This restores all pipeline definitions, notebook code, and Lakehouse schemas. Estimated time: 8-12 minutes for a workspace with 40-60 items.
  6. Rebind semantic models (T+40 minutes): Run the pre-written PowerShell script that calls the Power BI REST API to update all DirectLake semantic model connections from the primary Lakehouse item IDs to the DR Lakehouse item IDs. This script is stored in the Git repository under /dr-runbooks/rebind-semantic-models.ps1 and takes approximately 5 minutes for 12 semantic models.
  7. Update DNS and gateway routing (T+45 minutes): Update the Azure API Management policy routing analytics API traffic to point to the DR workspace endpoints. Update the on-premises data gateway configuration to connect to the DR capacity for any hybrid data sources.
  8. Validate and communicate (T+50 minutes): Run the DR validation notebook suite — 8 notebooks that check row counts against expected ranges, validate semantic model refresh, and confirm pipeline execution on the DR capacity. Send a stakeholder communication confirming analytics platform availability with a note on the 45-minute data lag window.
  9. Resume incremental pipelines (T+55 minutes): Trigger the incremental load pipelines manually for the highest-priority Gold layer tables. These pipelines use watermark-based incremental logic stored in a control table, so they automatically pick up from the last successfully processed batch without reprocessing historical data.
  10. Document and plan failback (T+60 minutes): Total RTO achieved: 55 minutes against a 2-hour target. Begin monitoring primary region recovery via Azure Service Health. Plan failback procedure for when East US capacity is restored, including a reconciliation step to merge any data written to DR storage during the outage window back into the primary OneLake.

Key Takeaways

  • Fabric's shared responsibility model places application-layer DR squarely on you: Microsoft protects infrastructure; you protect workspace configurations, pipeline definitions, semantic model metadata, and data integrity.
  • Delta time travel is your fastest recovery tool for data corruption, but only if you configure extended retention (delta.logRetentionDuration) and never run VACUUM with a retention period shorter than your RPO.
  • Git integration is necessary but not sufficient for DR: It backs up code and metadata, not data. A complete DR strategy requires both a code restore procedure (Git) and a data restore procedure (Delta restore or cross-region replication).
  • DirectLake semantic models have a hidden DR dependency on workspace item IDs — plan for a rebinding step in every failover runbook, and automate it with the Power BI REST API.
  • Multi-geo workspace support requires F64 or higher capacity — teams on F8 or F16 must use ADLS Gen2 shortcuts and manual replication pipelines as their cross-region DR mechanism.
  • Fabric Mirroring dramatically simplifies DR for operational source systems by maintaining a continuously updated replica in OneLake, eliminating the need for custom CDC pipelines for supported sources like Azure SQL Database and Cosmos DB.
  • Test your DR runbook quarterly with a real failover drill — a runbook that has never been executed is not a DR plan, it is a hypothesis.
  • Shortcut dependencies are the most commonly overlooked DR failure point — map every Shortcut to its source storage account and region before finalizing your DR architecture.
