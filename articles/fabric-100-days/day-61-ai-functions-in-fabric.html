Why AI Functions in Fabric Matter: The Gap Between Data and Intelligence

Every organization sitting on terabytes of structured data in OneLake eventually faces the same uncomfortable truth: the most valuable signals are buried in unstructured text. Customer support tickets, product reviews, contract clauses, medical notes, sales call transcripts — these assets exist in your Lakehouse but remain analytically invisible because traditional SQL and PySpark have no native vocabulary for language understanding.

Until recently, bridging this gap required a dedicated ML engineering team, Azure OpenAI provisioning, custom API wrappers, rate-limit handling, credential management, and a separate inference pipeline that had to be stitched back into your data estate. The operational overhead was substantial enough that most data teams simply deferred the work indefinitely.

Microsoft Fabric's AI Functions change this calculus entirely. Introduced as part of the Fabric AI capabilities roadmap, AI Functions expose large language model capabilities — sentiment analysis, entity extraction, summarization, classification, translation, and open-ended generation — as first-class SQL and PySpark functions that execute directly against data in your Lakehouse or Warehouse. No separate infrastructure. No API key juggling. No custom orchestration. The intelligence comes to the data, not the other way around.

This article covers everything a data engineer or analyst needs to know to use AI Functions effectively in production: the underlying architecture, the specific functions available, performance and cost considerations, and a complete real-world scenario you can replicate today.

Core Concepts: What AI Functions Actually Are

The Architectural Foundation

AI Functions in Microsoft Fabric are built on top of Azure OpenAI Service models, specifically GPT-4o and GPT-3.5-Turbo, provisioned and managed entirely within the Fabric platform. When you call an AI Function, Fabric routes the request through its internal AI gateway, which handles authentication via the workspace's managed identity, batches rows intelligently to maximize throughput, and returns results as a new column in your result set.

The critical architectural insight is that you never interact with Azure OpenAI directly. There is no endpoint URL to configure, no API key to store in Key Vault, and no deployment name to reference. Fabric abstracts all of this through its AI Functions layer. This has two important implications: first, your data never leaves the Fabric trust boundary during inference; second, the cost of inference is metered against your Fabric capacity (F-SKU) rather than billed separately through an Azure OpenAI subscription.

Where AI Functions Run

AI Functions are available in three Fabric compute contexts:

  • Fabric Notebooks (PySpark) — Using the synapse.ml library's OpenAIChatCompletion and dedicated AI Function wrappers, you can apply LLM operations to Spark DataFrames at scale across your Lakehouse Delta Parquet tables.
  • SQL Analytics Endpoint / Warehouse (T-SQL) — The ai_generate_text(), ai_summarize(), and related functions are callable directly in T-SQL SELECT statements against Warehouse tables or via the SQL Analytics Endpoint on Lakehouse tables.
  • Power Query (Dataflows Gen2) — A subset of AI Functions is exposed through Power Query's AI Insights connector, enabling no-code sentiment and key phrase extraction during data transformation.

Technical Deep Dive: Using AI Functions in Practice

T-SQL AI Functions in the Fabric Warehouse

The T-SQL AI Functions follow a consistent calling convention. The first argument is always the text input (a column reference or a string expression), and subsequent arguments are function-specific parameters. All functions require the Fabric Warehouse to be running on an F64 capacity or higher for production workloads, though F2 and F4 capacities can execute AI Functions with significantly reduced concurrency — expect throttling above 5 concurrent AI Function calls on F2.

-- Summarize customer support tickets and classify their urgency in a single pass
-- Assumes a Fabric Warehouse table: dbo.support_tickets
-- Columns: ticket_id (INT), ticket_text (VARCHAR(MAX)), created_date (DATE)

SELECT
    ticket_id,
    created_date,
    -- Summarize the ticket to 2 sentences maximum
    ai_summarize(ticket_text, 2) AS ticket_summary,
    -- Classify urgency into one of three predefined labels
    ai_classify(
        ticket_text,
        '["Critical - Service Down", "High - Feature Broken", "Low - General Inquiry"]'
    ) AS urgency_label,
    -- Extract structured entities: product name and error code
    JSON_VALUE(
        ai_extract(
            ticket_text,
            '{"product_name": "string", "error_code": "string"}'
        ),
        '$.product_name'
    ) AS extracted_product
FROM dbo.support_tickets
WHERE created_date >= DATEADD(DAY, -7, GETDATE())
  AND LEN(ticket_text) BETWEEN 50 AND 8000; -- Stay within token budget

A non-obvious but critical detail: ai_generate_text() and its siblings process rows sequentially within a single query execution unless you explicitly partition the workload. For tables with more than 10,000 rows, you should use a stored procedure with batching logic or, better yet, move the workload to PySpark where Spark's parallelism naturally distributes inference across executor nodes.

Also note the LEN(ticket_text) BETWEEN 50 AND 8000 filter. GPT-4o has a context window of 128K tokens, but Fabric's AI Functions impose a per-row input limit of approximately 8,000 characters (roughly 2,000 tokens) to protect shared capacity. Rows exceeding this limit return NULL with a warning in the query diagnostics, not an error — a silent failure mode that will corrupt your results if you don't filter proactively.

PySpark AI Functions in Fabric Notebooks

For large-scale batch inference against Delta Parquet tables in OneLake, the PySpark path using SynapseML's OpenAIChatCompletion transformer is the right tool. It distributes inference across Spark executors and handles retries and rate limiting automatically.

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, struct, lit, to_json
from synapse.ml.services.openai import OpenAIChatCompletion

spark = SparkSession.builder.getOrCreate()

# Load the last 30 days of product reviews from OneLake Delta table
reviews_df = spark.read.format("delta").load(
    "abfss://gold@onelake.dfs.fabric.microsoft.com//product_reviews.Lakehouse/Tables/reviews"
).filter(col("review_date") >= "2025-05-01").select(
    "review_id", "product_sku", "review_text", "star_rating"
)

# Build the prompt column — system + user message in OpenAI chat format
# SynapseML expects a column of type array>
from pyspark.sql.functions import array, create_map

def build_messages(text_col):
    """Returns a column expression producing the messages array for chat completion."""
    system_msg = struct(
        lit("system").alias("role"),
        lit(
            "You are a product analyst. Given a customer review, respond with a JSON object "
            "containing exactly two keys: 'sentiment' (one of: Positive, Neutral, Negative) "
            "and 'key_themes' (a list of up to 3 short phrases). Respond with JSON only."
        ).alias("content")
    )
    user_msg = struct(
        lit("user").alias("role"),
        text_col.alias("content")
    )
    return array(system_msg, user_msg)

reviews_with_prompt = reviews_df.withColumn(
    "messages", build_messages(col("review_text"))
)

# Configure the OpenAIChatCompletion transformer
# In Fabric, use the built-in service URL — no API key required
# The model deployment name maps to Fabric's internal GPT-4o endpoint
chat_transformer = (
    OpenAIChatCompletion()
    .setDeploymentName("gpt-4o")          # Fabric-managed deployment
    .setMessagesCol("messages")
    .setOutputCol("ai_response")
    .setMaxTokens(256)
    .setTemperature(0.0)                   # Deterministic output for classification
    .setErrorCol("ai_error")              # Capture errors per row instead of failing
)

# Run inference — Spark distributes rows across executors automatically
results_df = chat_transformer.transform(reviews_with_prompt)

# Parse the JSON response from the model
from pyspark.sql.functions import get_json_object, regexp_extract

parsed_df = results_df.withColumn(
    "ai_text", col("ai_response.choices")[0]["message"]["content"]
).withColumn(
    "ai_sentiment", get_json_object(col("ai_text"), "$.sentiment")
).withColumn(
    "ai_key_themes", get_json_object(col("ai_text"), "$.key_themes")
).select(
    "review_id", "product_sku", "star_rating",
    "ai_sentiment", "ai_key_themes", "ai_error"
)

# Write enriched results back to OneLake as a new Delta table
parsed_df.write.format("delta").mode("overwrite").option(
    "mergeSchema", "true"
).save(
    "abfss://gold@onelake.dfs.fabric.microsoft.com//product_reviews.Lakehouse/Tables/reviews_enriched"
)

print(f"Processed {parsed_df.count()} reviews. Errors: {parsed_df.filter(col('ai_error').isNotNull()).count()}")

The setErrorCol("ai_error") configuration is essential in production. Without it, a single malformed row or a transient throttling event will fail the entire Spark job. With it, failed rows are captured in the error column and the job continues — you can then inspect and reprocess failures separately.

Performance and Cost Considerations

Capacity Requirements by Workload Size

Workload Size Recommended SKU Estimated Throughput Typical Latency per Row Notes
Under 1,000 rows F2 / F4 ~5 rows/sec (T-SQL) 200–400ms Acceptable for ad-hoc analysis; avoid in scheduled pipelines
1,000–50,000 rows F16 / F32 ~20 rows/sec (T-SQL), ~80 rows/sec (PySpark) 50–200ms Use PySpark for this range; T-SQL becomes impractical above 10K rows
50,000–500,000 rows F64 ~200 rows/sec (PySpark, 8 executors) 20–80ms Partition input data; use incremental processing patterns
500,000+ rows F128 / F256 ~500+ rows/sec (PySpark, 16+ executors) 10–40ms Consider Delta Change Data Feed to process only new/changed rows

Best Practices

  • Filter before you infer: Apply all possible predicate filters (date ranges, non-null checks, length constraints) before rows reach an AI Function. Every row passed to the model consumes capacity CUs regardless of whether the output is useful.
  • Use temperature=0.0 for classification tasks: When using AI Functions for categorization, sentiment, or entity extraction, set temperature to 0 to get deterministic, reproducible outputs. Reserve higher temperatures only for creative generation tasks.
  • Store raw model output alongside parsed results: Always persist the raw JSON response from the model in your Delta table before parsing. This allows you to re-parse with updated logic without re-running expensive inference.
  • Implement incremental processing with Delta Change Data Feed: Enable delta.enableChangeDataFeed = true on your source tables and use readChangeFeed in your enrichment notebooks to process only new or modified rows, not the full table on every run.
  • Validate output schema with Great Expectations or Fabric DQ rules: LLM outputs are probabilistic. Even with temperature=0, the model occasionally returns malformed JSON or unexpected label values. Add schema validation as a post-processing step before writing to Gold layer tables.
  • Use system prompts to enforce output format: A well-crafted system prompt that explicitly specifies JSON output format, field names, and allowed values reduces parsing failures by 60–80% compared to relying on the user prompt alone.
  • Monitor CU consumption via Fabric Capacity Metrics app: AI Function calls are billed as Fabric CU-seconds. Open the Capacity Metrics app in your workspace and filter by "AI" operation type to track consumption before it impacts other workloads sharing the capacity.

Common Pitfalls

Pitfall Cause Fix
Silent NULL returns for long rows Input text exceeds Fabric's ~8,000 character per-row limit; function returns NULL without raising an error Add WHERE LEN(text_col) <= 8000 filter; for longer documents, chunk text into segments and aggregate results
Inconsistent classification labels Prompt does not constrain the model to a fixed label set; model invents synonyms like "Positive" vs "positive" vs "POSITIVE" Include explicit label list in system prompt and add a UPPER(TRIM(result)) normalization step post-inference
Spark job fails on throttling setErrorCol not configured; a 429 rate-limit response from the AI gateway throws an exception that kills the executor Always set .setErrorCol("ai_error") in SynapseML transformer; implement exponential backoff by reducing Spark parallelism with .repartition(8)
Re-processing entire table on each pipeline run Pipeline reads full Delta table instead of incremental changes, causing redundant and expensive inference on already-processed rows Enable Delta Change Data Feed on source table; use spark.read.format("delta").option("readChangeFeed", "true").option("startingVersion", last_version)
AI Function unavailable in SQL Analytics Endpoint T-SQL AI Functions are only available in Fabric Warehouse, not in the SQL Analytics Endpoint that sits on top of a Lakehouse Either migrate the table to a Fabric Warehouse, or use PySpark in a Notebook to run inference and write results back to the Lakehouse
Costs spike unexpectedly on F2 capacity AI Function calls on small capacities consume a disproportionate share of CUs due to the fixed overhead of model invocation, causing capacity throttling that spills into other workloads Schedule AI enrichment pipelines during off-peak hours using Fabric Pipeline scheduling; use burst capacity or upgrade to F16 for regular AI workloads

Real-World Scenario: Enriching a Customer Feedback Pipeline

A retail analytics team receives 15,000 new customer feedback records daily from three channels: e-commerce reviews, in-store survey responses, and contact center notes. The business needs sentiment scores, product category tags, and a one-sentence summary for each record to feed a Power BI dashboard used by category managers. Here is how they built this pipeline in Fabric:

  1. Ingest raw feedback into OneLake Bronze layer: Three Data Pipeline activities copy daily CSV exports from Azure Blob Storage into a Lakehouse Bronze Delta table (raw_feedback) using the Copy Data activity. Delta Change Data Feed is enabled on this table from day one: ALTER TABLE raw_feedback SET TBLPROPERTIES ('delta.enableChangeDataFeed' = 'true').
  2. Create a Silver cleansing notebook: A Fabric Notebook reads only new rows using readChangeFeed, deduplicates on feedback_id, removes records under 20 characters, and truncates text to 7,500 characters. Output is written to silver_feedback Delta table.
  3. Create an AI enrichment notebook: A second Fabric Notebook reads from silver_feedback using Change Data Feed. It applies the OpenAIChatCompletion transformer with a system prompt that requests a JSON response containing three fields: sentiment (Positive/Neutral/Negative), product_category (mapped to the company's 12 standard categories), and summary (one sentence, maximum 25 words). Temperature is set to 0.0. The raw model response is stored in ai_raw_response column.
  4. Parse and validate AI output: A post-processing step uses get_json_object() to extract the three fields. A validation check confirms sentiment is one of the three allowed values and product_category is in the master category list. Rows failing validation are written to a gold_feedback_quarantine table for manual review — typically fewer than 2% of daily volume.
  5. Write to Gold layer and refresh semantic model: Validated records are upserted into gold_feedback_enriched using Delta's MERGE statement on feedback_id. A final pipeline activity triggers a partial refresh of the Power BI semantic model using the XMLA endpoint, refreshing only the FeedbackFacts table to minimize refresh time.
  6. Monitor and alert: The team configured a Fabric Data Activator rule that fires a Teams notification if the daily quarantine rate exceeds 5%, which would indicate a prompt regression or an unexpected shift in feedback language patterns.

The end-to-end pipeline runs in approximately 18 minutes on an F32 capacity, processing 15,000 records at a cost of roughly 45 CU-minutes — well within the team's daily capacity budget and leaving ample headroom for interactive Power BI queries during business hours.

Key Takeaways

  • AI Functions in Fabric eliminate the need for separate Azure OpenAI provisioning by routing inference through Fabric's managed AI gateway, billed against your F-SKU capacity rather than a separate Azure subscription.
  • T-SQL AI Functions (ai_summarize(), ai_classify(), ai_extract()) are available only in Fabric Warehouse, not the SQL Analytics Endpoint — a common source of confusion when working with Lakehouse-backed tables.
  • The per-row input limit of approximately 8,000 characters is a silent failure: rows exceeding this limit return NULL without raising an error, making proactive filtering mandatory.
  • For workloads above 10,000 rows, PySpark with SynapseML's OpenAIChatCompletion transformer is dramatically more efficient than T-SQL AI Functions due to Spark's native parallelism across executor nodes.
  • Setting setErrorCol() in SynapseML transformers is non-negotiable in production — without it, transient throttling events will crash your entire Spark job rather than logging per-row failures.
  • Delta Change Data Feed is the correct pattern for incremental AI enrichment pipelines — processing only new or modified rows prevents redundant inference costs on already-enriched records.
  • Always persist raw model output before parsing; this allows schema evolution and re-parsing without re-running expensive inference against the model.
